# 0003: ゲートバイナリのセルフブートストラップ

- ステータス: 提案
- 日付: 2026-07-06
- 関連: 0002(配布物の責任境界・本決定の根拠)・#73(ゲート判定の Go 一本化)・#135(トークン化判定)・プリセット supply-chain-security(ピン留め検証の思想)

## コンテキスト

atelier のゲート判定は Go バイナリ(`atelier gate`)に一本化されている(#73・#135)。バイナリは OS/アーキテクチャ依存物であるため git で配布物に同梱できず、gitignore 下の `.agents/bin/` にマシンごとに配置する設計になっている。その取得手段が「fail-closed で停止したゲートのエラーメッセージに埋め込まれた 1 行を、人間が `!` プレフィックスで手動実行する」であり、次の問題があった。

- **デバイスごとの手動儀式**: `.atelier/`(管理下の印)は clone で全デバイスに付いてくるが、バイナリは付いてこない。新しいマシンで作業を始めるたびに、Bash が全停止した状態から手動復旧が要る
- **配布不能**: 配布先ユーザーには「プラグインを入れて init したら Bash が止まり、`gh` のインストールとログインを前提とした 1 行を貼らされる」体験になる。作者以外に配れない
- **検証の根が弱い**: 手動 1 行は「実行時点の最新リリースに付いている checksums.txt」を信用しており、リリースアセットの差し替えを検出できない

エコシステムの実態(2026-07 時点の調査): 公開されている Claude Code プラグインのフックは圧倒的に bash(+jq)であり、コンパイル済みバイナリを配布・ブートストラップする公開プラグインの前例は見つからなかった。これは大半のフックが単純な文字列検査で足りるためであり、コマンドのトークン化判定まで行う atelier のゲートは例外的に複雑なフックである。一方、Claude Code 本体は自分自身のバイナリを「マニフェストに記載した SHA256 チェックサムとの照合 + 署名」で配布しており、「ピン留めした検証値と照合してから実行する」方式には最も強い前例がある。

## 決定

プラグインのフックに、ゲートバイナリのセルフブートストラップを持たせる。

- **契機**: SessionStart フック。バイナリの欠落、または同梱ピンとのバージョン不一致を検知したときに取得する(ゲート発火時 PreToolUse のパスにはネットワーク I/O を混ぜない)
- **取得**: `curl` で GitHub Releases の公開 URL から取得する。`gh` CLI・GitHub 認証を前提にしない
- **検証**: プラグインに「バイナリのバージョン + checksums.txt」をピン留め同梱し、**同梱チェックサムと一致した場合のみ** `.agents/bin/` に配置する。信頼の根は、人間がレビューして commit したピンである(リリースアセット側の checksums.txt は信用しない)
- **バージョンスキューの解消**: 既存バイナリがピンと不一致の場合も取得・置換する。全マシンがプラグインのピンに収束し、更新配布も同じ機構で解決する。`.agents/` は gitignore 下の作業状態であり、置換は 0002 の範囲内
- **fail-closed の維持**: オフライン・タイムアウト・チェックサム不一致のときは配置せず、ゲートは現行どおり fail-closed で停止し、手動 1 行(従来手順)を案内する。ブートストラップは利便であって、防御の前提ではない
- **リリース手順**: バイナリのリリースごとに、プラグイン側のピン(バージョン + checksums.txt)を更新するコミットが必要になる。リリース workflow でピン更新を自動生成し、忘れを構造的に防ぐ

### 0002(配布物の責任境界)との整合

- 書き込み先は `.agents/bin/`(gitignore 下の作業状態)に限られ、「外部状態を持たない」に適合する
- ゲートは危険操作を**止める防御**であり、0002 が配布を許す側である。本決定は防御を成立させるための取得手段の整備であって、引き金の配布ではない

### 検討した代替案と退けた理由

- **bash への回帰**(バイナリ廃止): フックをスクリプトだけにすれば配布問題自体が消えるが、トークン化判定は bash では品質を保てない。文字列マッチ判定の事故(#135)から Go に移した経緯そのものへの退行であり、採らない
- **ランタイム前提のスクリプト言語同梱**(Python / TypeScript、MCP エコシステムの npx / uvx 型): 型安全・CI 検証可能でコードが git で届くが、配布問題が「ランタイムの存在という前提」に置き換わるだけで消えない。配布先に実行時依存を要求しない単一静的バイナリの利点を捨てる取引であり、採らない
- **バイナリを git リポジトリに commit**(プラグイン `bin/` 機構への同梱を含む): ブートストラップ自体が不要になるが、複数プラットフォーム × 毎リリースで git 履歴が肥大する。またプラグインの `bin/` 機構は正式ドキュメントのない挙動への依存になる。採らない
- **フックがリリースの checksums.txt で検証**(ピン非同梱): プラグイン更新なしで最新へ追従できるが、アセット差し替えに気づけず、ピン留めの意味がない。採らない

## トレードオフ

- **得るもの**: 配布可能性 — 設置体験が「プラグインを入れて init するだけ」になり、デバイス追加の儀式が消える。`gh` 非依存。検証がピン留め化され、現行の手動手順より強くなる。バージョンスキューの解消
- **諦めるもの**: リリースが 2 段になる(バイナリのリリース + プラグイン側のピン更新)。SessionStart にバイナリ欠落時のネットワーク I/O が入り、遅い回線ではセッション開始が待たされる
- **緩和策**: ピン更新はリリース workflow で自動生成する。取得は短いタイムアウトで諦め、失敗の帰結は「現行と同じ fail-closed + 手動案内」に閉じる
