# distributable-boundary: 配布物の責任境界

## 適用除外

配布物(プラグイン・CLI・パッケージ・サービス等、自分のリポジトリの外へ設置されて動く成果物)を作らないリポジトリには該当なし。

## コンテキスト

エージェント成果物は、成果物のアイデンティティ(プラグイン / CLI / サービス)ごとに、持ってよい副作用と配ってよい能力の範囲が異なる。この認識を欠いたまま器を固定して機能を積み増すと、2 種類の事故に至る。設置先マシンのグローバル状態(OS のスケジューラ・共有設定)へ書き込んで壊す事故と、実行主体(AI)が人間と同一の認証を共有する環境で、不可逆操作の便利な引き金が AI の届く場所に配られる事故である。いずれも実測された事故であり、根本原因は「機能要求から器を逆算せず、器の範囲を超える機能を積んだこと」に行き着く。

## 決定

### 1. 外部状態を持たない

- 配布物は自身の管理領域(設置先リポジトリ・明示された作業領域)の中にしか状態を書かない
- その外にあるマシンの状態 — OS のスケジューラ登録、グローバル設定、ホームディレクトリの共有ファイル — を変更しない
- マシン状態の設定が要る機能(定期起動など)は、**人間が設置するためのテンプレートの提供と、読み取り専用の診断**までに留める。設置そのものは人間の操作とする
- 例外: 実行基盤自身が管理する登録(配布機構のフック読み込み等)は、配布物のコードが書くものではなく、本決定の対象外

### 2. 共有認証下で危険な引き金を配らない

- 実行主体(AI)が人間と同一の認証を共有し、サーバー側が両者を区別できない前提では、不可逆操作(デプロイ・タグ作成・公開・削除)の「磨かれた便利コマンド」を配布物に含めない。引き金は人間が生のツールで直接引く
- 危険操作を**止める防御**(ゲート・検証)は配布してよい。配らないのは引き金であって、防御ではない
- 判断基準: そのコマンドは「人間の手数を減らす便利さ」を、AI が同じ認証で押せるリスクと引き換えにしていないか。していれば配布物から外し、手順書に置き換える

### 3. 機能から器を逆算する

- 新機能を設計するとき、まず「この器のアイデンティティ(ライフサイクルの紐づき先・認証の共有・全設置先での同一挙動)の範囲に収まるか」を問う
- 収まらないなら機能を捨てるか、**器を変える**(別のアイデンティティの成果物へ移す)。器を固定したまま範囲外の機能を積まない

### 4. 設置先 footprint は追跡可能・可逆にする

- 配布物が設置先リポジトリの commit 済みファイル(CI・テンプレート・設定・雛形)を設置する場合、置いた物を manifest(何を・どこに・撤去の逆再生方法)で追跡する
- 設置・撤去はどちらも reviewable かつ reversible な操作 — working tree の直接改変ではなく PR — で行い、冪等にする(再設置は差分ゼロなら no-op、撤去は manifest の逆再生で痕跡を残さない)
- 人間が後から足した記述を壊さぬよう、まるごと所有するファイルと、マーカー節 / API 状態(逆再生可能な単位)を区別する

## トレードオフ

- **得るもの**: 器の安全性 — 壊れた設定が利用者のマシンや復旧手段を巻き込まない。責務の明確化 — 配布物を消せば痕跡が残らない(設置先の committed 物も manifest 駆動の撤去 PR で可逆)。範囲外の機能を別アイデンティティへ切り出す判断が設計の最初から働く
- **諦めるもの**: 無人常駐・自動デプロイのような、器の外へ踏み出す機能は同じ配布物では提供できない。定期起動は人間の一手間(テンプレートの設置)を要する
- **緩和策**: 範囲外の機能は器ごと別プロダクトへ移す(この判断が本原則の主眼)。診断とテンプレート提供で人間の設置作業を最小化する
