# domain-modeling: ドメインモデルの原則

## 適用除外

業務ロジックを持たない領域(純粋な CRUD・変換・配線だけの層)には該当なし。ドメイン境界の切り方は domain-partitioning、永続スキーマの初期判断は db-design、契約は design-by-contract 相当のローカル規律が扱う。本 ADR は**その中間 — モデルそのものの形**を扱う。

## コンテキスト

モデルは物理的な実体の写しではなく、特定の目的を達成するための小さなシステムである。物理名詞(`User` / `Product` / `Account`)をそのまま 1 モデルにすると、あらゆる状況の属性とルールが集まって肥大し、目的の違うルールがフラグと分岐で衝突する。スキーマ(db-design)が強力なほど、エージェントは「既存テーブルからモデルを起こす」方向に流れやすい。モデルを目的から導く規律を上流に固定する。

## 決定

### モデルは物理実体の写しではなく目的の小さなシステム

- 同じ物理名詞でも、アクターと目的が違えば別モデルにする(例: 「商品」は審査品目 / 注文品 / 在庫品 / 配送品)。目的別モデルは目的外の属性・ルールを持たない
- 現実世界の名詞を機械的にクラス化しない。中心モデル(サービスの大通りにある `User` 等)の肥大は分割のシグナルとして扱う

### モデルは目的から導く。スキーマから起こさない

- 必要な属性・振る舞いは、アクター → 目的 → 目的達成を妨げる問題 → 必要な判断・制約 → 判断に必要な情報、の順に導く。既存テーブルや ER の列を先にモデルへ写さない
- スキーマ(db-design)はこのモデルの**下流**である。物理実体を 1 テーブル / 1 モデルへ素朴に対応させない

### 見えない概念を明示する

- 会話や条件式に埋もれた業務概念(資格・承認・予約・割当・上限・猶予・合意・請求・取消・判定・方針・権限)は、フラグや分岐ではなく**名前のある型・イベント・ポリシー**にする
- 重要な業務概念の意味をプリミティブ(素の string / int / bool)へ潰さない

### 何を入れないかを決める

- 各モデルに **out of scope(対象外)** を定義し、目的外の属性・操作を拒否できるようにする。これがモデルの単純さと変更容易性を守る

### 状況の違いはモデルの違いとして検討する

- 状態によって必要情報やルールが根本的に変わる場合(申請前と承認後・個人顧客と法人顧客・見積と契約)は、単一モデルの状態フラグではなく別の概念・役割・ライフサイクルを検討する
- 識別の一貫性を保つ必要がある一時的な遷移に限り、Entity 内の状態として扱う。目的・ルール・ライフサイクルの独立性で判断する

### 推測を事実にしない

- 目的や業務ルールをエージェントの一般常識で確定しない。推測は仮説として明示し、ドメインエキスパート(huddle)と具体例・反例で確認する。代表的な変更が一つの目的モデル内で完結することを検証する

## トレードオフ

- **得るもの**: 中心モデルの肥大とフラグ地獄を防ぎ、変更が一つの目的モデルに閉じる。暗黙の業務判断が名前のある概念として可視化され、レビューとテストの対象になる
- **諦めるもの**: 目的別モデルへ分けることで、同じ物理対象に複数のモデルが生まれる(意図した非統合 — domain-partitioning「同じ名詞を単一モデルへ統合しない」と対)。目的の見極めに人間の確認が要る
- **緩和策**: 純粋な CRUD・変換層には適用しない。小さな一時状態は Entity 内状態で足りる。分割の妥当性は「代表変更が 1 モデルに閉じるか」で検証し、最初から完璧な分割を狙わない
