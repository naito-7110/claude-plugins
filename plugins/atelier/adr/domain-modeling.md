# domain-modeling: ドメインモデルの原則

## 適用除外

業務ロジックを持たない領域(純粋な CRUD・変換・配線だけの層)には該当なし。ドメイン境界の切り方は domain-partitioning、永続スキーマの初期判断は db-design、契約は design-by-contract 相当のローカル規律が扱う。本 ADR は**その中間 — モデルそのものの形**を扱う。

## コンテキスト

モデルは物理的な実体の写しではなく、特定の目的を達成するための小さなシステムである。物理名詞(`User` / `Product` / `Account`)をそのまま 1 モデルにすると、あらゆる状況の属性とルールが集まって肥大し、目的の違うルールがフラグと分岐で衝突する。スキーマ(db-design)が強力なほど、エージェントは「既存テーブルからモデルを起こす」方向に流れやすい。モデルを目的から導く規律を上流に固定する。

## 決定

### モデルは物理実体の写しではなく目的の小さなシステム

- 同じ物理名詞でも、アクターと目的が違えば別モデルにする(例: 「商品」は審査品目 / 注文品 / 在庫品 / 配送品)。目的別モデルは目的外の属性・ルールを持たない
- 現実世界の名詞を機械的にクラス化しない。中心モデル(サービスの大通りにある `User` 等)の肥大は分割のシグナルとして扱う

### モデルは目的から導く。スキーマから起こさない

- 必要な属性・振る舞いは、アクター → 目的 → 目的達成を妨げる問題 → 必要な判断・制約 → 判断に必要な情報、の順に導く。既存テーブルや ER の列を先にモデルへ写さない
- スキーマ(db-design)はこのモデルの**下流**である。物理実体を 1 テーブル / 1 モデルへ素朴に対応させない

### 見えない概念を明示する

- 会話や条件式に埋もれた業務概念(資格・承認・予約・割当・上限・猶予・合意・請求・取消・判定・方針・権限)は、フラグや分岐ではなく**名前のある型・イベント・ポリシー**にする
- 重要な業務概念の意味をプリミティブ(素の string / int / bool)へ潰さない

### 不変条件をモデルの中心に据える

- モデルは業務データの箱ではなく、判断・計算・状態遷移・制約を所有し、不正な状態を作らせない仕組みとして設計する。守るべき不変条件は目的から抽出し、破ったときにどの目的・価値が損なわれるかを記述する
- 不正な状態を**表現不能または生成困難にする**: 生成時(コンストラクタ / ファクトリ)で検証し、public setter を避けて意味のある操作だけを公開する。制約付きの値は Value Object を検討する。検証を Controller / Service / 画面 / DB へ分散させない
- 制約の**権威ある所有者を一つに定める**(正の実装は一箇所、他層はそれを呼ぶ)。DB 制約は最後の防衛線として併用するが、業務ルールを DB だけに隠さない(db-design「整合性の砦は DB / 変わりやすい業務ルールはアプリ層」のアプリ側)
- **Aggregate / 整合性境界は不変条件から導く**: 「一緒に更新したい」ではなく「同一トランザクションで絶対に守る不変条件」でまとめる。結果整合でよいルールはイベント / プロセスで分離する(concurrency-process と接続)
- 失敗時は、不正な操作が拒否され状態が変わらず、部分更新・副作用(永続化・外部送信)が起きないことをテストで確認する。パターン名(Value Object 等)を先に当てはめない

### ロジックは状態の所有者へ集める

- 目的に強く関与するロジック(値の正しさの判断・状態変更・状態からの業務計算・遷移の許可 / 拒否)は、その状態を所有するモデルへ集める。データと振る舞いを別々の Util / Service へ散らさない(データとロジックの分離を疎結合と呼ばない)
- 自身の状態を使わず引数の別オブジェクトだけを操作するインスタンスメソッドは、所有者側へ移すか Policy / 純粋関数に分ける(Feature Envy)
- 情報を隠しても業務上の問いには答えられるようにする。内部コレクションや可変参照をそのまま返さず、意味のある query・immutable view・snapshot を返す
- カプセル化の単位は目的で決める(「同じテーブル / 同じデータだから」で束ねない)。ただし小ささを目的化しない — 一つの不変条件を守るのに多数のオブジェクトを巡回するなら分割しすぎ

### 何を入れないかを決める

- 各モデルに **out of scope(対象外)** を定義し、目的外の属性・操作を拒否できるようにする。これがモデルの単純さと変更容易性を守る

### 状況の違いはモデルの違いとして検討する

- 状態によって必要情報やルールが根本的に変わる場合(申請前と承認後・個人顧客と法人顧客・見積と契約)は、単一モデルの状態フラグではなく別の概念・役割・ライフサイクルを検討する
- 識別の一貫性を保つ必要がある一時的な遷移に限り、Entity 内の状態として扱う。目的・ルール・ライフサイクルの独立性で判断する

### 推測を事実にしない

- 目的や業務ルールをエージェントの一般常識で確定しない。推測は仮説として明示し、ドメインエキスパート(huddle)と具体例・反例で確認する。代表的な変更が一つの目的モデル内で完結することを検証する

### モデルの完全性を監査する

- 完全性とは、対象 scope で正しい判断と状態維持に必要な概念・制約・状態・遷移・振る舞い・失敗が揃い、重要な業務ルールがモデル外(Service / UI / DB trigger / 運用手順)へ漏れていないこと。現実の全情報をモデル化する意味ではない
- 名詞だけでなく**出来事**(注文された・取消された・期限切れになった)・**判断**(注文可能か・返金対象か)・**失敗**を抽出する。名詞中心はデータモデルになり振る舞いが漏れる
- **全 writer を列挙**し、同じ不変条件を通ることを確認する(API・batch・migration・admin tool・message consumer・test fixture・直接 SQL / import)。一入口の validation では別経路から不完全な状態を生成できる
- **全 reader を列挙**し、同じ概念を異なる意味で読んで表示都合でモデル意味を歪めていないか確認する
- **失敗を業務上区別すべき結果として型で表現**する(在庫不足・確定済み・権限不足・期限切れ・外部決済失敗)。外部技術失敗と業務不成立を混同しない
- 判断に時刻が影響するなら、暗黙に system clock を読まず、基準時刻の意味をモデルまたは契約で明示する
- 要件を、モデル要素・契約・テストへ追跡できるようにし、モデル外へ漏れた業務ルールを列挙・解消する(design-by-contract の追跡と対)。パターン名(DDD 等)の有無でなく、不正状態を生成できないか・重要ルールが漏れていないかで判定する

## トレードオフ

- **得るもの**: 中心モデルの肥大とフラグ地獄を防ぎ、変更が一つの目的モデルに閉じる。暗黙の業務判断が名前のある概念として可視化され、レビューとテストの対象になる
- **諦めるもの**: 目的別モデルへ分けることで、同じ物理対象に複数のモデルが生まれる(意図した非統合 — domain-partitioning「同じ名詞を単一モデルへ統合しない」と対)。目的の見極めに人間の確認が要る
- **緩和策**: 純粋な CRUD・変換層には適用しない。小さな一時状態は Entity 内状態で足りる。分割の妥当性は「代表変更が 1 モデルに閉じるか」で検証し、最初から完璧な分割を狙わない
