# technical-debt: 技術的負債の原則

## 適用除外

本プリセットは負債の解消・リファクタリングを**計画・実行するとき**にのみ効かせる(大規模移行の段階置換まで含む)。機能追加やバグ修正そのものには適用しない — 付随して負債に気づいた場合は即時 issue 化に回す(documentation / work)。

## コンテキスト

負債は「古いコード・長いクラス・警告件数」ではなく、目的に適したあるべき構造と現在の構造の差である。あるべき姿が未定義なら、何が負債か・どこまで直せば完了かを判定できず、エージェントは目につく巨大ファイルから無計画に着手しがちになる。限られた資源をどこへ振り、どこを意図的に直さないかを、記録される決定として固定する。

## 決定

### 負債はあるべき姿との差で定義する

- 負債の完了条件を「コードを整理した」ではなく**観測可能な状態**にする(例: 「割引ルールの変更が配送処理へ波及しない」)。着手前に、対象領域の目的・守るべき不変条件・責務を所有すべきモジュール・許容する依存方向・変更が完結すべき境界を定義する
- 負債を**因果連鎖**で記述する: 症状 → 破られている品質目標 / 不変条件 → 誤った責務所有者・境界 → 構造的原因 → 事業・開発への影響。症状だけで着手しない

### 兆候と証拠を分ける

- **兆候**(行数・複雑度・重複・引数数・低いテスト率)は着手の理由そのものにしない
- **設計上の証拠**(異なる目的の混在・不変条件の分散・誤ったレイヤ・意図しない公開・変更境界の崩壊)と**影響の証拠**(障害・変更失敗・長いリードタイム・複数チーム調整)で、直す理由と改善後の姿を確定する
- 静的解析の警告件数を、そのまま負債総量として扱わない

### do-not-fix を一級の決定にする

- 各項目を「今直す / 次に直す / 後回し / 直さない」に区別する。**後回しと do-not-fix は理由を記録する**(「なぜ直さないか」も憲法の一部 — adr プリセットの棄却 ADR と同じ思想)
- コアでも変化もない領域を、見栄えのために全面改修しない。変化も影響も小さい領域は無理に再設計しない

### 解消と再発防止を対にする

- 負債解消の issue は、**再発防止をセットで持つ**: 同じ責務混在を作らせない受け入れ条件・レビュー観点・characterization test(退行保護)。防止策のない解消を完了としない
- 上位項目には、挙動を維持する最小の安全な改善単位(段階置換)と、各段階の検証・停止条件を持たせる

### 最終優先順位は人間が持つ

- 事業重要度 × 今後の変更頻度 × 負債影響 × 失敗リスク ÷ 解消コストは、精密なスコアではなく**比較のための判断枠**である。根拠のない点数を作らず、High / Medium / Low と理由を残す
- 最終順位は product value と trade-off を判断できる**人間の owner** が決める。owner が不明なら確認方法を残す(fail-closed)

### 大規模リファクタリングの進め方

- **挙動を固定してから構造を変える**: 現行の外部観測挙動を characterization test で固定し、must-preserve(契約が依存) / intentional-change(明示的に直す) / unknown(意図未確認・勝手に変えない)に分類してから着手する(tdd-doctrine の状態検証で固定)
- **共通化は目的分割の後**: 先に DRY 化して異なる目的を共通層へ固定しない。目的別に分けて差異が見えた後、意味・契約・変更理由が同じものだけ抽出する(code-design と対)
- **安全な分割戦術**: 複雑な依存は「目的別の新しい入れ物へコピーし、その目的に不要な要素を削る」。内部依存が非常に複雑で変更リスクが高いときはストラングラーフィグで新旧並行置換にする。いずれも純粋計算を先に切り出し、代表的な一ユースケースを入力から出力まで通す垂直スライスで先に検証する(基盤だけ大量に作らない)
- **暫定経路の寿命**: コピー・adapter・feature flag・二重実装は移行手段であって完成形ではない。各暫定経路に owner・導入日・目的・利用を示す metric / log・削除条件・削除予定 phase を付け、**旧経路の利用がゼロで削除済みであることを完了条件にする**(feature-flags の期限・所有者を移行足場全般へ一般化)
- **停止条件(fail-closed)**: 維持すべき公開契約が特定できない / データの正の所有者を一つに決められない / 現行挙動を test・log・差分のいずれでも観測できない / 新旧並行で二重副作用を防げない / 不可逆なデータ変更に backup・dry-run・復旧手段がない、いずれかがあれば実装を進めず調査・計画へ戻す

## トレードオフ

- **得るもの**: 「巨大ファイルから順に着手」「行数削減を成果と称する」を防ぎ、負債の議論が事業重要度・変化・影響・リスクに沿う。do-not-fix が記録に残り、直さない判断を蒸し返さない。再発防止とセットにすることで負債の純増を止める。大規模リファクタリングが挙動固定・段階移行・旧経路削除まで含む安全な形になる
- **諦めるもの**: あるべき姿の定義と因果連鎖の記述という着手前コストがかかる。優先順位付けに人間の判断が要る。characterization test と垂直スライスの初期コストがかかる
- **緩和策**: 小さな負債は即時 issue 化に回し、本プリセットの重い手順は「解消をまとめて計画する」ときにだけ効かせる。優先度は精密計算でなく High / Medium / Low で足りる。分割戦術は垂直スライス 1 本から検証し、最初から全体を作らない
