# spec-alignment: 仕様すり合わせの原則

## 適用除外

なし。atelier の運用に乗るすべての issue に適用する。

## コンテキスト

仕様の曖昧さは実装の手戻りより高くつき、無人実行では事故になる。エージェント開発では issue の受け入れ条件が唯一の完了定義であり、「Ready にする」という行為の品質がすべての下流工程を決める。仕様確定は人間の責務(人間ゲート)であり、エージェントはその意思決定の場を整える。

## 決定

### Ready の条件

- **受け入れ条件が機械的に検証可能**(実行できるコマンド・観測できる挙動・「〜が green」の語彙)でない issue を Ready にしない
- **受け入れ条件が目的に接続されていること**: 「これが green だと、誰のどの困りごとが解けるか」を一文で言える。issue の完了定義が特定技術・特定クラスへ固定された手段の場合、それは候補手段として扱い、技術語を除いて問題を説明できることを確認してから Ready にする(手段の目的化を Ready として追認しない)
- スコープが閉じていること: 「今回やること / やらないこと」が明確で、外部の決定待ち・他 issue への暗黙依存がないこと(依存は `依存: #N` で明示)
- 変更境界が明示されていること: 触ってよい範囲(may_change: 内部実装・内部データ構造・局所的な命名)と、触ってはいけない範囲(must_preserve: 公開契約・データ意味・既存利用者の観測可能な挙動 / must_not_change: 認可方針・課金計算・未承認スキーマ)、対象外(out_of_scope)を区別する。自律実行するエージェントの安全柵として、やること / やらないこととは別軸で持つ

### 仕様すり合わせの必須アジェンダ

issue の内容にかかわらず、次を必ず扱う(該当しない項は「該当なし」を確認して高速に閉じる)。第 1〜2 項は設計レベルの前段ゲート(目的・品質)で、これを通してから手段レベル(3 以降)を確定する:

1. **目的・手段の分離**: この issue の完了定義が「目的」か「手段」かを判定する。手段先行(特定技術・クラスへの固定)なら候補手段として退避し、「つまり何のためか」で一段上の目的へ遡る。技術語を除いて問題を説明できない場合は Ready にしない。タイトルが素直に目的を表す明白なタスクは「目的＝タイトル、OK」で即通過してよい
2. **品質 portfolio の確定**: この変更で積極的に上げる品質を 1〜3 個に絞る(primary)。全品質の同時最大化を狙わない。意図的に最適化しない品質があれば明記する(「ここは下限を守るだけで最速は目指さない」)。守るべき下限は各領域プリセット(performance / resilience / product-security ほか)が既に定義済みなので再掲しない。品質が明白なタスクは「primary＝機能正確性、OK」で即通過してよい
3. **排他制御の選択**(concurrency-process): 書き込みを伴うか、方式はどちらか
4. **feature flag の要否**(feature-flags): 必要なら name / owner / 期限をその場で決める
5. **最小 PR の分割案**(pr-granularity): 依存順つきの分割案を合意する
6. **API リソース設計**(rest-api-design。REST の場合): エンドポイントを伴うならメソッド × パス・ネスト構造を確定する
7. **マージレーンの提案**(merge-policy): agent マージ可か人間マージかを提案し、人間が承認する
8. **依存の追加**(supply-chain-security): 新規に必要となる依存(ライブラリ・ツール)を洗い出して issue に明記する。選定は pros/cons を添える。ここに明記されていない依存が実装中に増えた場合は agent マージ失格(merge-policy)
9. **文書への影響**(documentation): この変更で古くなる文書(ドメイン知識・公開契約・地図・スタック事実)を特定し、**同 PR での更新を受け入れ条件に含める**

### 確定事項の昇格

- すり合わせで確定した内容は issue 本文へ**「確定済みの設計(日付 + 出所)」**として書き戻す。以後の実装はこれを仕様として扱う
- issue 本文の編集権はすり合わせの場に限定する(トリアージ・実装セッションは提案コメントまで)

### 曖昧なまま進まない

- 曖昧・矛盾した仕様で実装に入らない。着手後に発覚した場合は作業を止め、選択肢と pros/cons を添えてエスカレーションする(fail-closed)

## トレードオフ

- **得るもの**: 手戻りの最小化・無人実行の安全・完了判定の客観性。下流(実装・レビュー・マージ)の判断がすべて issue 本文だけで完結する
- **諦めるもの**: 着手前の対話コスト。小さなタスクにも一定の儀式がかかる
- **緩和策**: 明白なタスクは必須アジェンダを「該当なし」の連打で数分で閉じる。アジェンダを固定化しているのはこの高速化のため(何を確認すべきかを毎回考えない)
