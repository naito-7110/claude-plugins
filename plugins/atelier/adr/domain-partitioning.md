# domain-partitioning: ドメイン分割の原則

## 適用除外

なし。単一ドメインで回る小規模リポジトリは「全体 = 1 ドメイン」または `domains: {}` のままでよく、**分割しないという判断も本プリセットの適用**である(分割は義務ではない)。

## コンテキスト

ドメイン分割の質が、所有マップの正しさ・ドメイン文書の鮮度・ドメインエキスパート(huddle)の判断品質をすべて決める。技術層で切った分割は「あらゆる変更が全ドメインを跨ぐ」構造を生み、独立評価も所有も機能しなくなる。分割の判断基準を憲法として固定する。

## 決定

### 分割の単位

- **変更理由・業務能力の単位で切る**: 「誰の・どの業務要求で変わるか」が同じものを 1 ドメインにまとめる
- **技術層(フロントエンド / バックエンド / DB)で切らない**。技術層はすべてのドメインを貫く断面であり、ドメインではない

### 境界の条件

- ドメイン間は**公開契約(contracts)だけで会話する**。他ドメインの内部(非公開のモジュール・テーブル・実装詳細)に直接手を入れない・依存しない
- 契約に無いやり取りが必要になったら、契約の追加(該当ドメインの決定)として扱う

### 所有の排他性

- **1 パス 1 ドメイン**。同じパスを複数ドメインが所有しない(機械検証の対象)
- 共有物(共通ライブラリ・ビルド基盤)は、共有ドメイン(例: `shared`)として明示的に宣言するか、いずれかのドメインの所有に寄せて契約で提供する。「なんとなく共有」を作らない

### 同じ名詞を単一モデルへ統合しない

- 同じ名詞・同じ ID が複数ドメインに現れても、1 つの共通モデルへ統合しない。アクターと目的が違えば必要な属性・ルール・状態遷移・ライフサイクルは異なる(例: 「商品」は注文品 / 配送品 / 在庫品)。各ドメインは自分の目的に必要な意味・ルール・状態を所有する
- ドメイン間で共有してよいのは**安定した識別子と明示的な公開契約だけ**。可変な業務ルール・目的固有の状態・計算を共有モデルへ寄せない
- 「共通」という名前のドメインへ業務ルールを集めない。DRY を理由にした多目的モデルの共有は、変更が複数ドメインへ波及する原因になる(粒度シグナルの「小さすぎ」ではなく「多目的化」の兆候)

### 粒度のシグナル

- **大きすぎる兆候**: ドメイン内の無関係な箇所へ変更が波及し始める・ドメイン文書が肥大して読まれなくなる
- **小さすぎる兆候**: ほとんどの変更が複数ドメインを跨ぐ・契約変更が頻発する
- どちらの兆候も再編(分割・統合)の検討材料であり、放置しない

### 再編は決定として記録する

- 分割・統合・パスの移管は、**起点の issue / PR を参照する決定**(該当ドメインの decisions またはローカル ADR)として記録し、所有マップの変更と同じ PR で行う(documentation の出所原則)

### 導入は漸進

- 全体を一度に切らない。境界が見えている 1 ドメインから宣言を始め、残りは未所有(警告)のままでよい

### 設計投資の濃淡

- 全ドメインを同じ設計密度で作らない。差別化を生むドメイン(コア)には深いモデル・強い制約・手厚いテストを投資し、汎用・補助ドメインは build/buy/reuse と単純実装を優先する。変化も影響も小さい領域は無理に再設計しない
- どのドメインがコアかの判定はプロダクト戦略であり **人間が決める**。所有マップのドメインに設計密度の意図(例: `core` / `supporting` / `generic`)を任意注記として持たせ、変更するときは再編と同じく決定として記録する
- コアに集中投資できるよう、汎用ルールの共通化でコア固有の概念を薄めない(過剰共通化の禁止)。「重要な基盤だから」という理由だけで技術基盤をコア扱いしない

## トレードオフ

- **得るもの**: 変更が原則 1 ドメインに閉じ、独立評価(huddle)・所有・文書の鮮度管理が機能する。分割の議論が「好みの構造」ではなく判断基準に沿って行える。設計資源を差別化領域へ集中でき、汎用領域の過剰設計を防ぐ
- **諦めるもの**: 業務能力での分割は初期の見極めコストが高い(技術層分割の方が最初は簡単に見える)。再編に決定の記録という手続きがかかる。コア度の見極めという人間判断も初期にかかる
- **緩和策**: 粒度シグナルを再編のトリガーにすることで、最初の分割を「完璧」にする必要をなくす(間違えたら記録付きで直す)。小規模な単一ドメイン repo は設計密度を一律(generic)としてよく、濃淡は分割が要る規模でのみ効かせる
