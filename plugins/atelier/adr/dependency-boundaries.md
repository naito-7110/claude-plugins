# dependency-boundaries: 依存境界の原則

## 適用除外

コードを持たないリポジトリ(文書のみ等)には該当なし。単一ファイル・単一モジュールで完結する小規模コードにも、第 2 決定(内部は具象のまま)がそのまま適用される — 「抽象化しない」という判断も本プリセットの適用である。

## コンテキスト

エージェントは抽象化を「全部に適用」か「全く適用しない」の両極に振れやすい。前者はどこにも実体のない間接層の霧を生み(読む者が実装へ辿り着けない)、後者は外部 I/O が奥深くまで染み込んで差し替え・テスト不能を生む。境界の引き方には判断基準が要り、それは規約(善意)ではなく可能な限り構造で強制されるべきである。なお本プリセットが扱うのは**モジュール・コードレベル**の依存境界であり、ドメイン間の境界(公開契約)は domain-partitioning が扱う(高度の異なる補完関係)。

## 決定

### 1. 境界は抽象(interface)に依存する

- 抽象を置くのは次の場所に限る: **実装が複数あり得る場所・外部 I/O との接続面・テストで差し替えたい場所**
- コア(ドメインロジック)は具体実装を知らない。依存の向きは常に「具象が抽象に依存する」方向に揃える

### 2. 内部は具象のまま

- 実装が 1 つしかあり得ない内部まで抽象化しない。**「2 つ目の実装が現れたら昇格」を既定の運用**とし、将来の可能性だけを理由に間接層を先置きしない
- 判断基準: その抽象を消したとき、テストか差し替えが不可能になるか。ならないなら、その抽象は霧である

### 3. 強制は構造で

- モジュール境界の強制は、可能なら言語・ビルド機構に載せる(パッケージ・モジュール分割で依存方向をコンパイル時 / ビルド時に検査させる)。レビュー規約(善意)より構造(機械検証)を優先する
- 構造で強制できないスタックでは、依存方向の静的検査をリンター等で CI に載せることを次善とする

### 4. 型・状態による分岐は目的別実装へ分ける

- すべての if / switch を消そうとしない。まず分岐の意味を分類する: 種類ごとの振る舞い(strategy / polymorphism 候補)・ライフサイクル中に変わる状態(state 候補)・業務ルール表(rule object 候補)・機能フラグ(feature-flags プリセット)・一時的なガードや単純変換(if のまま明快)
- 同じ種類分岐が複数箇所へ重複し、種類追加のたびに既存分岐を編集する場合に限り、種類固有のデータと振る舞いを一つの実装へまとめ、**利用側の目的で命名した契約**の背後へ隠す(`process()` / `execute()` のような空虚な契約にしない)。呼び出し側が type code を読み続ける状態を残さない
- 実装の選択責任は変更頻度の低い組み立て境界(factory / DI / registry)へ集約する。選択器を新たな業務ルールの集積地にしない
- 外から交換する方針(strategy)と、オブジェクト自身の遷移で変わる状態(state)を区別する。状態を strategy として外部管理して遷移の所有者を失わない
- 第 2 決定(2 つ目の実装で昇格・霧の禁止)はここにも適用する。単一実装で変化のない分岐を interface 化しない。言語の sealed type / pattern matching の方が契約を明確にできることもある

## トレードオフ

- **得るもの**: テスト容易性・差し替え可能性・境界侵犯の機械検出。エージェントの設計判断に「どこに抽象を置くか」の基準が入り、両極への振れを止められる
- **諦めるもの**: 境界の初期設計コスト。パッケージ分割等の構造的強制は最初の一手が重い
- **緩和策**: 昇格ルール(2 つ目の実装で抽象化)により、初期は最小の境界(外部 I/O 接続面のみ)から始めて育てられる
