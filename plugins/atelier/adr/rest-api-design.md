# rest-api-design: REST API リソース設計

## 適用除外

HTTP API を公開しないリポジトリには該当なし。**本プリセットは REST 専用**であり、他のスタイル(gRPC・GraphQL 等)の API には適用しない(採用する場合は対応する原則を別プリセットまたはローカル ADR として定める)。内部向けでも REST API を持つなら適用する。

## コンテキスト

URL・メソッドの設計を実装ごとの判断に任せると、RPC 風のパスとリソース指向が混在して API 全体の一貫性が失われる。設計の揺れは並行開発のコンフリクトと、クライアント側の場当たり対応も生む。REST を採用する場合の規約として、機械的に従える形に固定する。

## 決定

### リソース指向・動詞禁止

- **URL に動詞を入れない**(`POST /users/{id}/update` のような RPC 風パスは禁止)。リソース指向の複数形名詞を基本とする
- **URL セグメントはケバブケースに固定する**(URL の可読性を優先)。コード側の命名から機械的に変換できる形を保つ

### CRUD の対応を固定

| ユースケース | ルート |
| --- | --- |
| 一覧 | `GET /api/users` |
| 取得 | `GET /api/users/{userId}` |
| 作成 | `POST /api/users` |
| 更新 | `PATCH /api/users/{userId}`(全置換のみ `PUT`) |
| 削除 | `DELETE /api/users/{userId}` |

### 親子リソース(ネスト)

- **ライフサイクルが親に従属する(親なしで存在しない)リソースはネストで表現する**: `GET /users/{userId}/orders`・`POST /users/{userId}/orders`。親子関係は URL パスで与え、**リクエストボディに親 ID を持たせない** — 関係の指定が URL 側に寄ることで、ボディはリソース本体の属性だけになり単純に保てる
- **ネストは 1 段まで**。子リソースが自前の ID を持つなら、2 段目以降はトップレベルの正準 URL で直接参照する(`GET /orders/{orderId}`)。`/a/{aId}/b/{bId}/c/{cId}` のような深いパスを作らない
- **独立したライフサイクルを持つ・複数の親から参照されるリソースはトップレベル**に置き、絞り込みはクエリパラメータで表現する(`GET /orders?userId=...`)
- **多対多の関係は、関係そのものをサブリソースとして表現する**: 付与 = `PUT /users/{userId}/roles/{roleId}`、剥奪 = `DELETE /users/{userId}/roles/{roleId}`
- ネストかトップレベルかの判断基準は「**親なしで存在しうるか**」。迷うケースは仕様すり合わせのリソース設計アジェンダで確定する

### CRUD に収まらない操作

- 動詞ではなく**サブリソース(名詞)**として設計する。例: 有効化は `POST /api/users/{userId}/activation`
- こうしたリソース設計は**仕様すり合わせの必須アジェンダ**として着手前に確定させる(spec-alignment と接続)

### 互換性と deprecation

- **破壊的変更の定義**(merge-policy の失格条件「破壊的な外部 API 変更」はこの定義で判定する): フィールド・エンドポイントの削除やリネーム、型・意味の変更、必須パラメータの追加、エラー契約の変更、enum 値の削除
- **追加的変更は非破壊**とする: 新フィールド・新エンドポイント・任意パラメータ・enum 値の追加。クライアントは未知のフィールドを無視して動く前提(tolerant reader)で実装する
- 破壊的変更は避けるのが既定。必要な場合は「新しいバージョン/リソースを並行提供 → クライアント移行 → 旧の deprecation(猶予期間と告知)→ 削除」の順で行う(schema-migration の expand-contract と同型)
- deprecation は応答・仕様上で機械的に判別できる形で示し、削除は猶予期間の後に別リリースで行う

### 変更が閉じる配置

- 1 ユースケースの追加・変更の diff が専用の領域(ディレクトリ等)に閉じる構成を選ぶ。エンドポイントを共有ファイルに集中させない(pr-granularity・並行開発と接続)

## トレードオフ

- **得るもの**: API 全体が一貫し、ルーティングが規約から機械的に定まる。親子関係が URL に載ることでボディが単純になり、ユースケース追加の diff が閉じて並行開発のコンフリクトが最小になる
- **諦めるもの**: ファイル・ディレクトリ数が増え、全エンドポイントの一覧性が下がる。深い階層のドメインでは正準 URL(トップレベル)とネスト URL の使い分けの判断が都度必要になる
- **緩和策**: 一覧性は API 仕様(OpenAPI 等)の自動生成で担保する。使い分けの迷いは「親なしで存在しうるか」基準と仕様すり合わせで吸収する
