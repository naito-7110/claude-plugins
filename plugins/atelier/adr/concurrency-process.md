# concurrency-process: 排他制御の選択プロセス

## 適用除外

同時更新が起こらないリポジトリ(読み取り専用・単一書き手のバッチ等)には該当なし。複数の書き手が同じデータを更新しうるなら適用する。

## コンテキスト

同時更新による更新の喪失(lost update)と、check-then-act 型の競合への対処を各ユースケースの実装判断に任せると、方式も外部契約(ヘッダー・応答)も揺れる。方式の技術的な実装はスタックに依存するため、ここでは**選択のプロセスと既定**だけを固定する。

## 決定

### 既定は楽観ロック

- 既定は**楽観ロック**: エンティティにバージョン識別子を持たせ、更新時にバージョン一致を条件とする。競合は「既に変更済み」カテゴリの失敗値で返す(error-handling の AlreadyModified 相当)
- **版数管理を自作しない**: バージョンの比較・競合検出は、採用した DB・データアクセス基盤に付随する仕組み(同時実行トークン・条件付き更新)を使う。自作実装はバグの温床になりやすい(具体の機構はスタック事実・レシピ側)
- HTTP API の場合は web 標準の契約に統一する: バージョンを ETag で公開し、更新は If-Match で受け、競合は 412 を返す

### トランザクション境界はユースケース層の責務

- トランザクションの開始・コミット・ロールバックの境界は**ユースケース層**が持つ。ドメイン層は境界を知らず、境界(API)層はトランザクションを開始しない
- 1 ユースケース = 1 トランザクションを既定とし、それで表現できない処理(長時間処理・外部システムを跨ぐ整合)は仕様すり合わせで扱いを確定する

### 悲観ロックは限定使用

- 悲観ロック(行ロック・アプリケーションロック等)は **check-then-act 型の不変条件**(残数を確認してから引き当てる類、競合の発生自体を許せない処理)に限定する

### 選択はプロセスルール

- **どちらを使うかは、ユースケース単位で仕様すり合わせ(grooming)の場で人間が決定する**。エージェントが独断でロック方式を決めない
- その際、競合しうる操作・データ・同時利用シナリオ(影響範囲)と各方式の pros/cons を提示した上で選ぶ
- 書き込みを伴わないユースケースは「該当なし」を確認して閉じる(spec-alignment の必須アジェンダ)

## トレードオフ

- **得るもの**: 外部契約と競合時の挙動が全体で 1 パターンに統一される。方式選択が人間ゲートを通るため、ロックの乱用・誤用が構造的に防がれる
- **諦めるもの**: 楽観ロックはクライアント側に再取得・再送の実装を要求する。悲観ロックはロック待ちによるスループット低下の恐れがある(既定を楽観にすることで乱用を防ぐ)
- **緩和策**: 仕様すり合わせのアジェンダに組み込むことで、選択の手間を着手前の 1 回に閉じる
