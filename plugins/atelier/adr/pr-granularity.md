# pr-granularity: PR 粒度の原則

## 適用除外

なし。PR を作るすべての作業に適用する。

## コンテキスト

atelier では人間のレビューがマージゲートであり、「人間がレビューできる PR」であることがスループットの生命線になる。並行するエージェント開発ではコンフリクトの最小化も効く。粒度の基準をサイズではなく関心事に置き、判断を機械的にできるようにする。

## 決定

### 分割基準は「関心事が単一で最小か」

- コミット・PR の分割基準は**関心事の単一性**であり、サイズ(行数・ファイル数)ではない
- **機械的な一括変更**(単一の変数名の一括置換・一括フォーマットなど、読んでいて苦にならないもの)は何ファイルあっても 1 PR でよい
- ただし**複数種の一括変更の混在**(複数の変数名の置換を 1 PR に混ぜる等)はレビュー負荷が跳ね上がるため、種類ごとに分割する
- **独立してレビュー・マージできる成果物が複数**含まれるなら、小さくても分割する(例: 独立した部品 3 つ → 3 サブ issue = 3 PR)。分割案は issue コメントで提案・合意する
- **「ついでのリファクタリング」を機能変更に混ぜない**。小さな diff でも関心事が 2 つになった時点で粒度は大きい(リファクタは別 PR に切り出す)
- **迷ったら小さく**: 小さく出して後から足す方が、大きく出して削るより常に安い

### サイズはシグナルとして使う

- サイズ超過(目安: 20 ファイル / 800 行)は「関心事が混ざっていないか」を見直すシグナル。単一関心だと説明できるなら超えてよい(PR の概要にその旨を明記する)

### 分割は PR ではなく issue で行う(1 issue = 1 PR)

- **「1 issue = 1 PR」を不変条件とする**。1 つの issue を複数の PR に分けない(stacked PR = 1 issue に PR を積む運用は扱わない)
- 実装の分割が必要になったら、PR を積むのではなく、仕様すり合わせの場が**サブ issue に分割**する。各サブ issue は本文の `依存: #N` 行で依存順を明示する(issue 単位の依存検査と、マージ時の Closes による自動クローズがそのまま正しく働く)
- PR は必ず対応する 1 つの issue を Closes で紐づけて閉じる(途中 PR を Refs で残す運用を作らない)

### 積み上げの作法

- 未完成の機能は feature-flags の背後に置き、main を常にリリース可能に保ったまま、成果物・レイヤーごとにサブ issue(= PR)を積む
- 分割したサブ issue には依存順を明示する(`依存: #N` 行)。依存の管理は PR 間ではなく issue 間で行う

## トレードオフ

- **得るもの**: レビュー時間の短縮と精度の向上・並行開発でのコンフリクト最小化・revert の容易さ。issue 単位の依存検査・自動クローズと粒度管理が一本化される
- **諦めるもの**: issue と PR の数が増え、依存順の管理と積み上げの手数がかかる。1 変更内の途中成果を PR として積む自由を失う
- **緩和策**: 分割案の提案を仕様すり合わせ(spec-alignment)の必須アジェンダに含め、着手前に切り方を確定する
