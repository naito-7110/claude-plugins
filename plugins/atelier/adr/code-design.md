# code-design: コード設計の原則

## 適用除外

コードを持たないリポジトリ(文書のみ等)には該当なし。命名の機械的規約(テーブルはスネークケース・複数形、URL はケバブケース、ログ属性は OpenTelemetry 準拠)は db-design / rest-api-design / logging-observability が持つ。本 ADR はそれらの下敷きになる**目的駆動の設計判断**を扱う。

## コンテキスト

名前は可読性の飾りではなく、開発者の認知・責務のまとまり・境界を方向づける。曖昧な名前は異なる目的の処理を同じ場所へ集め、置いてよいものと置いてはいけないものの区別を失わせる。エージェントは既存名や一般知識に引っ張られた命名をしがちで、その名前がそのまま責務の集積地になる。名前を目的から導く判断基準を固定する。

## 決定

### 名前は目的から導く

- 名前は「誰が使うか・何を達成するか・何を所有するか・何を保証するか・何を持たないか」から導く。既存の名詞・クラス名・DB カラム名に引っ張られない
- 名詞は文脈内の役割、command は実現する操作、query は答える内容、event は完了した事実(過去形)、boolean は肯定形で真の意味が明確になるようにする。技術的手順ではなく業務上の結果を名前へ優先する

### 曖昧な名前を責務診断に使う

- `Manager` / `Service` / `Util` / `Helper` / `Common` / `Base` / `Data` / `Info`、`process` / `handle` / `execute`、`And` / `Or` を含む名前、極端に長い名前は、責務が隠れている・複数責務が混在しているシグナルとして扱う。フレームワーク慣例名や明確な Application Service まで機械的に否定はしない
- 説明に「〜もする」「場合によっては」が続く名前は、命名ではなく責務分割の兆候

### 名前は適用範囲を狭める

- 良い名前は「ここへ足してよいもの」と同時に「足してはいけないもの」を示す。候補ごとに in-scope と out-of-scope を確認し、境界として機能するかを見る

### アンカリングを避け、Rename か分割かを判断する

- 既存名を一時的に伏せ、ユースケースと目的から名前候補を出してから現在名と比較する
- 目的に合わせた名前の下でメンバ(field / method)を fits / does-not-fit に分類する。合わない要素が多ければ Rename で済ませず責務分割へ進む(名前で説明しきれないのは分割の兆候)
- 用語はコード・テスト・API・イベント・会話で一貫させる(documentation のユビキタス言語一貫性と対)。参照の機械的更新の安全性(IDE / language server を使い、文字列検索置換に委ねない)は別途 refactoring 時の規律で扱う

## トレードオフ

- **得るもの**: 名前が責務境界として働き、何をどこへ置くかの判断とレビューが名前だけで下せる。曖昧な集積地(`Manager` / `Util` など)の増殖を止める
- **諦めるもの**: 命名にアクター・目的の確認コストがかかる
- **緩和策**: フレームワーク慣例名は尊重する。明白な命名は候補検討を省略してよい。名前は長さではなく意味と境界の明確さで評価する
