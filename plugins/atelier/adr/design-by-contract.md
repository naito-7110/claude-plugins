# design-by-contract: 契約による設計の原則

## 適用除外

公開操作の契約(事前・事後・不変条件)を持たない領域(純粋な参照・変換・配線)には該当なし。不変条件のモデル所有・不正状態の排除は domain-modeling、契約テストの検証様式と破壊観点は tdd-doctrine が持つ。本 ADR は**要件を検証可能な契約へ変換し追跡可能にする**判断を扱う。

## コンテキスト

生成 AI(エージェント)は同じ要求でも異なる内部構造・命名を選ぶ。実装の形を固定しても再現性は得られない。契約(事前条件・事後条件・不変条件・失敗時の状態)を固定すれば、実装が違っても「無効入力を受理しないか / 正常時に求める結果を返すか / 失敗時に状態を壊さないか / どの公開操作でも不変条件を保つか」を機械的に判定できる。ここでの再現性は「毎回同じコード」ではなく **毎回同じ契約を満たすコード** である。

## 決定

### 要件を検証可能な契約へ変換する

- 「適切に検証する」「エラーを考慮する」ではなく、観測可能な文にする(例: 数量が 1 未満または 200 超なら注文を生成しない / 残高不足時は決済記録と注文状態を変更しない / 同じ冪等キーの再要求は同じ結果を返し二重決済しない)
- 公開操作ごとに、事前条件・事後条件・維持する不変条件・失敗時の状態・根拠(要件 ID)を持つ契約として整理する。要件に根拠のない条件は推測せず、unknown と仮定を分ける
- 契約はメソッド単位に限らず、Value Object・Entity / Aggregate・ユースケース・API endpoint・Repository・message / event・バッチのレベルで作れる

### 条件を分類する

- 呼び出し側が満たすべき → 事前条件、処理が保証する → 事後条件、モデルが常時守る → 不変条件、外部依存の可用性など → 環境条件。**環境条件をドメイン不変条件へ混ぜない**
- **事前条件と入力サニタイズを混同しない**。入力の早期検証は UX とエラーメッセージのためであり、契約の事前条件とは別

### 契約の所有者を一つに定める

- 値そのものの妥当性 → Value Object、集約内の整合性 → Aggregate、順序 / 権限 / 外部連携 → Application service、永続化形式 → Repository / schema、通信形式 → API / message 境界。同じ契約を複数箇所へ重複実装しない(不変条件のモデル所有は domain-modeling と対)
- 入力 DTO の validation だけでドメイン不変条件を保証したことにしない。別経路からモデルを生成すると破られる

### 契約テストを実装より先に、契約でトレースする

- 実装前に、事前条件の境界前後・正常時の事後条件・失敗後に変更されない状態・全公開操作後の不変条件・許可 / 禁止遷移・(重要なら)冪等 / 重複 / 順序を検証するテストを設計する(検証様式と破壊観点は tdd-doctrine と対)
- 各要件が少なくとも一つの契約項目とテストへ**追跡できる**ようにする。実装の適合は AI の説明ではなく、テスト結果とコード上の責務配置で判定する

## トレードオフ

- **得るもの**: 実装が変わっても要件適合を機械判定でき、エージェントが生成するコードのばらつきに再現性の土台が入る。失敗時の状態破壊を見落とさない
- **諦めるもの**: 要件を契約へ変換し、契約表と追跡を作る着手前コスト
- **緩和策**: 重い契約は重要な公開操作・状態変更に絞る。単純な参照や機械的変更には課さない。契約項目とテストの追跡は受け入れ条件の一部として扱う(spec-alignment と接続)
