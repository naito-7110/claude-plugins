# ai-usage: AI 利用の境界と責務

## 適用除外

製品にも開発フローにも AI(LLM 等)を使わないリポジトリには該当なし。atelier で開発する時点で、少なくとも「開発フロー」と「生成物」の節は適用される。

## コンテキスト

AI 時代の攻撃面は 3 つある: **AI が生成したコードそのもの**(品質のばらつきと幻覚)、**AI を組み込んだアプリケーション**(自然言語が新しい注入経路になる)、**開発環境の AI エージェント**(開発者マシン・資格情報が標的になる)。いずれも従来の枠(OWASP 等)だけでは捉えきれないため、境界と責務を憲法として固定する。

## 決定

### AI が生成した成果物

- **同一基準**: AI が生成したコード・設定・文書も、人間が書いたものと**同一のレビュー・テスト基準**を通す。「AI が書いたから」は品質・セキュリティの免責にならない
- **AI の出力を AI 以外の手段でも検証する**: 決定論的な検査(型・lint・テスト・SAST/SCA・ポリシー検証)を基盤に置き、AI レビュアは併用に留める(AI が AI の宿題を採点する構図に依存しない)
- **幻覚を前提にする**: 実在しない依存・API・権限アクションを自信を持って提案しうる。依存は実在確認(dependency-licensing)、権限・設定は検証機能に通し、過剰権限(ワイルドカードリソース等)を既定で疑う
- **再現性は同じコードでなく同じ契約の充足**: 生成物の評価は、コードの形やスタイルの一致ではなく、承認済みの問題・要件・契約・公開境界を満たすかで行う(design-by-contract と対)。同じ要求を複数回生成して比較するときは、コード差や最短さでなく、要件適合・契約テスト・不正状態の生成可否・品質シナリオ・変更局所性で選ぶ。生成担当と検証担当を分け、生成時の説明ではなく実行結果と追跡可能性で判定する

### 変更作業における役割分担

- **機械的で完全性を要する変更(Rename・Move・Extract・参照更新・import 整理)は IDE / language server / compiler が行う。AI に広範囲の文字列置換を委ねない**。AI に機械変更を依頼する場合も、参照の完全性は言語ツールで確認する
- **AI の出力は設計上の仮説**として扱う。目的・責務・制約の分析には根拠(path / symbol / test / 文書)・反証・confidence を要求し、コードから目的を断定させない。曖昧なレガシーでは複数解釈を残し、不足する事業文脈を AI に補完させない
- **人間が決めることを AI に委ねない**: 誰のどの目的を優先するか・何を品質改善とみなすか・どの既存挙動を契約として維持するか・破壊的変更の許可・提案の採否。「きれいにして」ではなく改善の定義(目的・品質目標・維持挙動・対象外)を先に与える
- **実装とテストの自己正当化を防ぐ**: 契約・受け入れ条件は人間が確認した仕様から先に定義し、characterization test は変更前の観測から作る。実装後の差分は別の prompt / 別 agent で検査する(決定論的検査を基盤に、AI が AI を採点する構図に依存しない)。テストが実装内部でなく契約を検証するか、mutation / 境界 / 負ケースで感度を確認する
- 変更は一 step 一目的で読める差分に閉じ、各 step を build・型検査・テストで即検証し、失敗時に戻せるようにする(pr-granularity・technical-debt の段階移行と対)

### 製品に AI を組み込む場合

- **データ境界**: 秘匿情報(secrets・資格情報)と個人データを外部 AI サービスに送らない。送る必要がある場合はマスキング・契約・同意の扱いを仕様すり合わせで確定する(data-lifecycle と接続)
- **prompt injection を前提に設計する**: データと命令の混同という点で従来のインジェクションと同型だが、**パラメータ化クエリに相当する完全な防御策が現存しない**。従って:
  - システム命令とユーザー入力を構造的に分離する(入力をシステムプロンプトへ文字列合成しない)
  - **LLM の出力も信頼できない入力**として扱う(描画前のサニタイズ・構造化利用時のスキーマ検証)
  - 注入経路は直接入力だけでなく、**AI が読む外部コンテンツ(間接)と保存された設定・記憶(蓄積)**も含めて扱う
  - LLM に与えるツール・権限は許可リストで最小化し、影響の大きい操作は人間の承認を挟む(fail-closed)
- **耐性の回帰テスト**: 完全防御が無い以上、注入攻撃のシミュレーションを CI の回帰テストとして持ち、モデル・プロンプトの変更で耐性が劣化していないかを継続検証する
- **モデル・プロンプトの変更は挙動変更**としてレビュー対象にする(暗黙のデプロイをしない)

### 開発環境の AI エージェント

- **エージェント・MCP サーバー・スキル等の導入は依存の導入と同じ扱い**にする(supply-chain-security / dependency-licensing の選定基準: 公式性・実績・透明性)。スキル定義・プロンプト等の**自然言語資産も注入経路**であり、コードと同じレビュー対象とする
- **エージェントの権限を最小化する**: 秘匿ファイル(環境変数ファイル・鍵・クラウド資格情報)への読み取りを拒否し、ネットワークは許可リスト、影響の大きい操作は人間承認。サンドボックス(ファイルシステム・ネットワーク分離)を多層防御の一層として使う
- **漏洩前提設計**: 「読ませない」だけに頼らず、開発資格情報は平文ファイルからシークレットマネージャへ移し、短命トークン・最小スコープにする(infrastructure の一時権限と接続)。標準出力・ログ経由の間接流出(デバッグ出力に偽装した窃取)も経路として認識する

## トレードオフ

- **得るもの**: AI 起点の 3 つの攻撃面(生成物・組み込み・開発環境)に既定の防御姿勢が入り、エージェント自律性を上げても被害半径が制御される
- **諦めるもの**: AI 機能・エージェント導入の手数が増える。prompt injection は完全には防げないという前提を受け入れる(残余リスクの明示)
- **緩和策**: 決定論的検査・許可リスト・回帰テストで「昨日防げた攻撃が今日通る」変動を検知可能にする
