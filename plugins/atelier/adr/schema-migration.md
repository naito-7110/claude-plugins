# schema-migration: スキーマ変更とデータ移行の原則

## 適用除外

永続化スキーマ(RDB・スキーマを持つデータストア)を持たないリポジトリには該当なし。

## コンテキスト

スキーマ変更は「本番でしか失敗が顕在化せず、ロールバックが最も難しい」変更である。merge-policy はスキーマ変更を常に人間マージに隔離しているが、**安全な書き方の原則が無ければ人間レビューにも判断根拠が無い**。エージェントがマイグレーションを書く前提で、進め方を型に固定する。

## 決定

### expand-contract(並行変更)を既定とする

スキーマ変更は 3 段階に分けて進める:

1. **expand**: 後方互換な追加(新しい列・テーブル・null 許容での追加)だけを先行リリースする
2. **migrate**: 新旧が併存する期間に、書き込みの二重化またはバックフィルでデータを移行する
3. **contract**: 旧要素の削除・制約の強化(NOT NULL 化等)は、全コードが新要素へ移行して安定した後、**別リリース**で行う

### 後方互換の原則(N-1 互換)

- **稼働中の旧バージョンのコードが新スキーマで動くこと**を各段階の条件とする(デプロイ中の新旧併存・ロールバックの両方がこれで成立する)
- 列の削除・リネーム・型変更・NOT NULL 化は破壊的変更であり、contract 段階でのみ行う。リネームは「新列追加 → 移行 → 旧列削除」に分解する

### スキーマ変更とコード変更を束ねない

- スキーマ変更(マイグレーション)とアプリケーションコードの変更を**同一デプロイに束ねない**。順序は「スキーマ先行(expand)→ コード → contract」

### マイグレーションの書き方

- **冪等・再開可能**に書く: 途中失敗からの再実行で壊れない(resilience の原則と接続)
- 大量データのバックフィルはバッチ化し、進捗を記録して再開できる形にする。ロック時間の長い一括操作を本番に流さない(performance の上限原則と接続)
- マイグレーションは実 DB のテストで検証する(tdd-doctrine)

### ロールバック計画必須

- すべてのスキーマ変更に戻し方を定義する。**down が書けない変更(データ破棄を伴う)は、事前バックアップと復元手順の確認をリリース条件**とする(infrastructure の可逆性・data-lifecycle の復元検証と接続)

## トレードオフ

- **得るもの**: 無停止でのスキーマ変更・任意の時点でのロールバック可能性・新旧併存への耐性。エージェントのマイグレーション PR に対する人間レビューの判断根拠
- **諦めるもの**: リリース回数が増え、移行期間中は新旧の二重管理が発生する
- **緩和策**: 小さな変更(null 許容の列追加のみ等)は expand 1 段で完結する。3 段階が要るのは破壊的変更を含むときだけ
