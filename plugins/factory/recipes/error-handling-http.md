# error-handling-http: 意味カテゴリと HTTP の正準カタログ

[error-handling](../adr/error-handling.md) プリセットの「意味カテゴリ」の、HTTP トランスポートにおける正準定義。HTTP API を持つプロジェクトが対象。カテゴリの追加・判定基準の追記は本レシピの改訂(通常 PR)で行う。

## 意味カテゴリと HTTP ステータス

ユースケース層が返す失敗値のカテゴリと、変換カタログ(1 箇所)での対応:

| カテゴリ | 意味 | HTTP |
| --- | --- | --- |
| NotFound | 対象が存在しない | 404 |
| AlreadyModified | 楽観ロックの競合(concurrency-process) | 412 |
| Forbidden | 認可エラー(権限不足。authorization) | 403 |
| Unauthenticated | 認証済みでない(基準は authorization) | 401 |
| ValidationFailed | 入力検証エラー | 422 |
| Conflict | 業務上の競合(重複登録など) | 409 |
| Unexpected | 予期しない失敗(ドメイン例外の到達など) | 500 |

## 境界(API)層が直接返すもの(ユースケース層は関与しない)

失敗値が生まれる前に境界層・ミドルウェアが直接返す応答。カテゴリのカタログには含めない:

| ステータス | 発生源 |
| --- | --- |
| 400 BadRequest | 構文レベルの不正(壊れた JSON・バインド失敗)。意味レベルの不正は 422 |
| 401 | 認証ミドルウェア |
| 403(粗い権限) | 認可の境界層(authorization の 1 段目) |
| 429 | レートリミット(product-security) |

## 判定ガイド(揺れやすいケース)

| ケース | 判定 |
| --- | --- |
| URL が指す対象が存在しない | NotFound |
| ボディ内の参照先が存在しない(選択済み ID の削除など) | ValidationFailed + 該当フィールドのエラー |
| If-Match のバージョン不一致 | AlreadyModified(412 は条件付きリクエスト専用) |
| 業務上の重複・衝突(同名の重複作成など) | Conflict |

**主語ルール**: 判定に迷ったら「エラーの主語」で切り分ける。主語がリクエスト内容(入力とサーバー状態の不整合)なら ValidationFailed(422)、主語がリソースの現在状態(内容は妥当だが状態が操作を許さない)なら Conflict(409)。RFC 9110 の定義(409 = 対象リソースの現在状態との競合、422 = 構文は正しいが内容を処理できない)に整合する。

## 応答形状

- エラー応答は RFC 9457(Problem Details)準拠を推奨。検証エラーはフィールド別エラーの拡張で返す

## フロントエンドとの接続

- フロントエンドは失敗値の契約(安定コード + カテゴリ)を解釈してユーザビリティ対応(再取得の誘導・フォームへのエラー表示・フォールバック)を組む。目的が異なるため、本カタログの様式をフロントエンド内部のエラー処理に強制しない
