# i18n-typescript: TypeScript での文言の型強制

[i18n-copy](../adr/i18n-copy.md) プリセットの「キーと文言の欠落は機械検査で検出する」を、TypeScript プロジェクトで**コンパイル時に強制する**ための手法集(非規範)。

## キー型の導出

- 原文言語のリソース(例: `ja.json`)を**型のソース**にする: リソースを `as const` 相当で取り込み、キー集合の型を導出する。翻訳ファイルではなく原文言語だけが型の真実
- ネストしたリソースはテンプレートリテラル型で `"domain.section.key"` 形式に平坦化して型化する

## 型付き t() の強制

- 文言参照は型付き `t()` ラッパー経由に限定する: `t(key: MessageKey, params: ParamsOf<Key>)`。**存在しないキーはコンパイルエラー**になる
- 補間パラメータは、メッセージ文字列からテンプレートリテラル型でプレースホルダー(`{name}` 等)を抽出し、`params` の必須プロパティとして型に反映する。**パラメータの渡し忘れ・余剰もコンパイルエラー**
- リソースオブジェクトへの直アクセスと、文字列連結による動的キー生成を lint で禁止する。動的にキーを選ぶ必要がある場合は、キーの union 型を引数に取る関数にして網羅性を保つ

## 適合テスト(CI)

- 追加言語のキー集合が原文言語と**一致**することをテストで検査する(欠落と余剰の両方向)
- プレースホルダー集合が言語間で一致することも検査する(翻訳時の `{name}` 抜けを検出)

## ハードコード検出

- UI コード内の生文言リテラルを lint ルールで検出する(JSX のテキスト・属性へのリテラル記述を禁止する系のルール)。例外(記号・単位など)は許可リストで管理する

## ライブラリの選定基準

- 上記が満たせるなら自作の薄いラッパーで足りる(型付き `t()` は小規模で書ける)。既製ライブラリを使う場合も「**キー型の導出・パラメータ型検査・適合テストのしやすさ**」の 3 点が効くことを採用条件にする。どれを採用するかの選定はローカル ADR に記録する
