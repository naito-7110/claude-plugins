# api-resource-design: API リソース設計

## 適用除外

HTTP API を公開しないリポジトリには該当なし。内部向けでも HTTP API を持つなら適用する。

## コンテキスト

URL・メソッドの設計を実装ごとの判断に任せると、RPC 風のパスとリソース指向が混在して API 全体の一貫性が失われる。設計の揺れは並行開発のコンフリクトと、クライアント側の場当たり対応も生む。機械的に従える規約に固定する。

## 決定

### リソース指向・動詞禁止

- **URL に動詞を入れない**(`POST /users/{id}/update` のような RPC 風パスは禁止)。リソース指向の複数形名詞を基本とする
- URL セグメントの命名はケバブケースなど 1 つの流儀に固定し、コード側の命名から機械的に変換できる形を保つ

### CRUD の対応を固定

| ユースケース | ルート |
| --- | --- |
| 一覧 | `GET /api/users` |
| 取得 | `GET /api/users/{userId}` |
| 作成 | `POST /api/users` |
| 更新 | `PATCH /api/users/{userId}`(全置換のみ `PUT`) |
| 削除 | `DELETE /api/users/{userId}` |

### CRUD に収まらない操作

- 動詞ではなく**サブリソース(名詞)**として設計する。例: 有効化は `POST /api/users/{userId}/activation`
- こうしたリソース設計は**仕様すり合わせの必須アジェンダ**として着手前に確定させる(spec-alignment と接続)

### 変更が閉じる配置

- 1 ユースケースの追加・変更の diff が専用の領域(ディレクトリ等)に閉じる構成を選ぶ。エンドポイントを共有ファイルに集中させない(pr-granularity・並行開発と接続)

## トレードオフ

- **得るもの**: API 全体が一貫し、ルーティングが規約から機械的に定まる。ユースケース追加の diff が閉じ、並行開発のコンフリクトが最小になる
- **諦めるもの**: ファイル・ディレクトリ数が増え、全エンドポイントの一覧性が下がる
- **緩和策**: 一覧性は API 仕様(OpenAPI 等)の自動生成で担保する
