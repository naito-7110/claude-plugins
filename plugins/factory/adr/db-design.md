# db-design: DB 設計の原則

## 適用除外

永続データストアを持たないリポジトリには該当なし。RDB 前提の項(制約・正規化)は RDB 採用時のみ。周辺の責務は分担済み: 変更手順は schema-migration、排他とトランザクション境界は concurrency-process、分類・保持・削除要求は data-lifecycle、index と読み取り上限は performance。本 ADR は**設計そのものの初期判断**を扱う。

## コンテキスト

スキーマはデータが乗った瞬間に変更コストが跳ね上がる。ID の形式・整合性の置き場・削除の方式・型の選択は「後から直すのが最も高くつく判断」であり、エージェントが設計する前提では既定の答えを固定しておく必要がある。変更に強い設計の核は「**変わらない事実を保存する**」ことにある。

## 決定

### データストアの選択

- **既定は RDBMS**。制約・トランザクション・アドホックな問い合わせ・運用知見の蓄積があり、要件が固まりきらない段階の既定として最も外しにくい
- 別種のストア(ドキュメント・KV・全文検索・時系列等)の採用は、**RDB で困る具体的な理由**(アクセスパターン・データ構造・スケール要求)を示した上で技術選定としてローカル ADR に記録する。併用(ポリグロット)は同期の責務とセットでのみ
- 半構造化カラム(JSON 型等)は「スキーマが定まらない・検索条件にしない」属性に限って許容する。検索・結合・制約の対象になる業務データは列に昇格させる(正規化の原則)
- 以降の項のうち制約・正規化は RDB 前提。**事実の保存・追跡方式・ID・型の規律はストアを問わず適用する**

### 事実を保存する

- **変わらない事実を保存し、導出できる情報は保存しない**(生年月日は事実、年齢はその時点の導出情報 — 計算で得る)
- **上書き更新は事実の消失**である。業務上意味のある変化(状態遷移・経緯)は、行の上書きではなく**事実の追記・状態テーブルの分離**で表現することを検討する
- 履歴・監査ログを現役のテーブルに混ぜない(履歴テーブルまたは専用基盤へ分離する)

### 状態追跡とイベント追跡

- データの持ち方には **状態追跡**(現在の状態を行として保持し、変化は更新)と **イベント追跡**(起きた事実を追記し、状態は導出する)の 2 方式がある
- **既定は状態追跡 + 必要な範囲の履歴分離**。ただし「**経緯そのものが業務上の事実**」である領域(取引・承認・残高や在庫の増減など)はイベント追跡(追記型)を採り、上書きで事実を消さない
- 選択の観点: 監査・経緯復元の要求があるか / 遡及訂正・再計算が要るか / 導出した状態(読み取りモデル)の維持コストを払えるか。ユースケース単位で仕様すり合わせで確定する(排他制御の選択と同じ場)

### 1 テーブル 1 責務とライフサイクル分割

- **1 テーブル 1 責務**。単純なテーブルほど変更に強い
- **更新タイミング(ライフサイクル)が異なる属性は別テーブルに分割する**(登録時に確定する属性と、運用中に変わる属性を同居させない)
- 責務肥大のシグナル: 多数の index が必要になる・NULL の多い列が増える。列を許容する前にテーブル分割を検討する

### 整合性の真実は DB

- 不変条件のうちデータベースで表現できるもの(NOT NULL・一意制約・外部キー・CHECK)は**DB の制約で強制する**。アプリケーション層の検証は入力の UX とエラーメッセージのためであり、整合性の最後の砦は DB(「真実はサーバー」と同型の思想)
- 「アプリ側が守るから制約は不要」を禁止する。並行実行・バグ・手作業のデータアクセスは制約しか防げない
- ただし制約に載せるのは**安定した構造的不変条件**(存在・一意・参照・値域)まで。変わりやすい業務ルール(状態遷移の可否など)は変更容易なアプリケーション層に置き、DB に埋め込まない

### ID 戦略

- **外部に見せる ID**: 推測不能で、かつ生成順が時系列に並ぶ識別子(UUIDv7 相当)を使う。連番を外部に公開しない(存在と件数が漏れ、列挙攻撃の標的になる)
- 外部に見せない内部の関係行は連番でよい
- 人間向けの表示コード(注文番号など)は識別子と分離し、ドメインの関心事として別途設計する

### 正規化と非正規化

- **正規化を既定**とする。重複保持(非正規化・キャッシュ列)は計測に基づく明示的な判断とし、理由と同期の責務(いつ・どう再計算するか)をローカル ADR に記録する(performance の計測駆動と同じ規律)

### 削除の方針

- **既定は物理削除**
- ビジネス上意味のある「削除済み」を残す場合も、**削除フラグ(deleted_at 等)ではなく、削除済み(アーカイブ)テーブルへの移行**を採る。フラグ方式は現役テーブルが肥大し続け、すべてのクエリに除外条件が伝播し、index の効率も落ちるため
- 個人データの削除可能性は data-lifecycle の要求が優先する

### 型と値の規律

- 時刻は **UTC で保存**し、タイムゾーンは表示層の関心事とする
- 金額・厳密な精度が要る数値を**浮動小数点型で保存しない**(固定小数点または最小単位の整数)
- **NULL に「不明・未設定」以外の業務的意味を載せない**。意味のある状態は明示の値・状態テーブルで表現する(NULL が増えるならライフサイクル分割のシグナル)

### 命名

- **既定: テーブル・列は小文字のスネークケース、テーブル名は複数形**。コード側の命名から機械的に変換できる形を保つ(変更はローカル ADR の Overrides で)

## トレードオフ

- **得るもの**: 「後から直せない判断」が既定で安全側に倒れる。事実の保存とライフサイクル分割により、仕様変更がテーブル追加(変更に強い形)で受けられる。制約が DB にあることで、実装バグがデータ破損に到達しない
- **諦めるもの**: 制約・正規化・分割の初期設計コスト。テーブル数は増える
- **緩和策**: プロジェクト固有の例外(非正規化・アーカイブ方式の変更等)は理由つきのローカル ADR と `Overrides` で逃がす
