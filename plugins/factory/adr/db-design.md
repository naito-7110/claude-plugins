# db-design: DB 設計の原則

## 適用除外

永続データストアを持たないリポジトリには該当なし。RDB 前提の項(制約・正規化)は RDB 採用時のみ。周辺の責務は分担済み: 変更手順は schema-migration、排他とトランザクション境界は concurrency-process、分類・保持・削除要求は data-lifecycle、index と読み取り上限は performance。本 ADR は**設計そのものの初期判断**を扱う。

## コンテキスト

スキーマはデータが乗った瞬間に変更コストが跳ね上がる。ID の形式・整合性の置き場・削除の方式・型の選択は「後から直すのが最も高くつく判断」であり、エージェントが設計する前提では既定の答えを固定しておく必要がある。

## 決定

### 整合性の真実は DB

- 不変条件のうちデータベースで表現できるもの(NOT NULL・一意制約・外部キー・CHECK)は**DB の制約で強制する**。アプリケーション層の検証は入力の UX とエラーメッセージのためであり、整合性の最後の砦は DB(「真実はサーバー」と同型の思想)
- 「アプリ側が守るから制約は不要」を禁止する。並行実行・バグ・手作業のデータアクセスは制約しか防げない

### ID 戦略

- **外部に見せる ID**: 推測不能で、かつ生成順が時系列に並ぶ識別子(UUIDv7 相当)を使う。連番を外部に公開しない(存在と件数が漏れ、列挙攻撃の標的になる)
- 外部に見せない内部の関係行は連番でよい
- 人間向けの表示コード(注文番号など)は識別子と分離し、ドメインの関心事として別途設計する

### 正規化と非正規化

- **正規化を既定**とする。重複保持(非正規化・キャッシュ列)は計測に基づく明示的な判断とし、理由と同期の責務(いつ・どう再計算するか)をローカル ADR に記録する(performance の計測駆動と同じ規律)

### 削除の方針

- **既定は物理削除**。論理削除(削除フラグ)は「すべてのクエリに除外条件が伝播し、一意制約・外部キーと衝突する」構造的負債を買う判断なので、要件(復元・参照整合の維持)を仕様すり合わせで確認した場合のみ採る
- 監査・履歴が目的なら、現役の行を汚す論理削除ではなく履歴(イベント)テーブルの分離を先に検討する
- 個人データの削除可能性は data-lifecycle の要求が優先する

### 型と値の規律

- 時刻は **UTC で保存**し、タイムゾーンは表示層の関心事とする
- 金額・厳密な精度が要る数値を**浮動小数点型で保存しない**(固定小数点または最小単位の整数)
- **NULL に「不明・未設定」以外の業務的意味を載せない**。意味のある状態は明示の値・状態列で表現する

### 命名の一貫性

- テーブル・列の命名の流儀(単数/複数・区切り・大文字小文字)はプロジェクトで 1 つに固定し、コード側の命名から機械的に変換できる形を保つ(具体の流儀はローカル ADR)

## トレードオフ

- **得るもの**: 「後から直せない判断」が既定で安全側に倒れる。制約が DB にあることで、エージェントの実装バグがデータ破損に到達しない。外部 ID から内部情報が漏れない
- **諦めるもの**: 制約・正規化の初期設計コスト。論理削除に慣れた運用からの転換コスト
- **緩和策**: プロジェクト固有の例外(非正規化・論理削除の採用等)は理由つきのローカル ADR と `Overrides` で逃がす
