# product-security: 開発物のセキュリティ

## 適用除外

すべてのプロダクトコードに適用する。UI・HTTP API を持たないリポジトリでは該当する項のみ適用する(入力検証・secrets・エスカレーション範囲は常に適用)。

## コンテキスト

セキュリティは個別機能の実装時に判断すると漏れが生じる横断関心事であり、factory ではコードの大半をエージェントが書く。「何を既定で有効にし、何をエージェントの裁量から外すか」を憲法として固定する(開発環境側は supply-chain-security)。

## 決定

### 境界での入力検証

- 入力検証を受信側の境界で必須とする。クライアント側の検証はユーザビリティのためであり、**検証の真実は常にサーバー(実行側)**
- 検証失敗は error-handling の意味カテゴリ(ValidationFailed 相当)へ正規化する

### 定番攻撃への既定(OWASP 上位)

- **SQL インジェクション**: 既定は ORM 経由のクエリとし、生 SQL が必要な場合も必ずプリペアドステートメント(パラメータ化)を使う。**外部入力の文字列結合によるクエリ組み立てを禁止**する
- **インジェクション全般**: 外部入力を OS コマンド・ファイルパス・クエリへ文字列結合で渡さない。コマンド実行は引数を配列で渡し、シェル文字列を組み立てない。ファイルパスは正規化と許可範囲の検証を挟む(パストラバーサル対策)
- **XSS・出力エスケープ**: フレームワークの自動エスケープに任せ、raw 出力(エスケープ無効化)は原則禁止。必要な場合はサニタイズ + レビュー必須
- **SSRF**: サーバー側から外部入力由来の URL へリクエストする場合は、許可リストで宛先を制限する
- 認可はサーバーで強制する(詳細は authorization プリセット。クライアント側の制御は防御ではない)
- 通信・cookie・セキュリティヘッダは安全側の既定を有効にする(具体の設定値はスタック事実・レシピ側)

### 認証チケットの保管

- 認証チケット(セッション・アクセストークン)を **JavaScript から到達できる保管場所(localStorage / sessionStorage 等)に置かない**。既定は HttpOnly + Secure + SameSite の cookie、またはトークンをサーバー側に留めてブラウザには不透明なセッションだけを渡す構成(BFF)
- **長寿命の資格情報(リフレッシュトークン)をブラウザの JS 到達領域に置かない**。SPA で使う場合はトークン更新エンドポイント限定の HttpOnly cookie + ローテーション + 再利用検知を必須とする
- **ネイティブクライアント**: リフレッシュトークンはプラットフォームのセキュア保管(OS のキーチェーン・キーストア相当)に置く。平文ファイル・共有ストレージ・アプリ設定への保存を禁止する
- **bearer 提示には所持証明を必須とする**: トークンを Authorization ヘッダー等で bearer として直接提示する構成(JWT のアクセストークン等)を採る場合は、**sender-constraining(DPoP 等の所持証明)を必須**とし、盗まれたトークン単独でのリプレイを成立させない。HttpOnly cookie で運ぶ構成では cookie 側の防御(SameSite・Secure)が代替するため対象外
- **ブラウザ向け Web アプリケーションの認証に JWT を既定としない**: 既定は不透明なセッション識別子 + HttpOnly cookie。JWT の採用が正当なのは、バックエンド間の認証(マイクロサービス・BFF の内側)とネイティブクライアントに限る
- **JWT を使う場合、クレームに意味のある情報を持たせない**: 個人情報(メールアドレス等)や権限の詳細をクレームに入れず、中身は最小限の不透明な識別子(subject 等)に留める。JWT は署名されているが**暗号化されておらず誰でも読める**こと、発行後の内容は失効まで古びる(権限変更が即時反映されない)ことを前提に設計する
- クライアントが持つチケットは**最小権限・短命**にする(infrastructure の一時権限と同じ思想)
- この保管規律の効果は **XSS 被害の縮小(資格情報の持ち出しと攻撃の持続化の阻止)であり、XSS 対策の代替ではない**。ページが生きている間のセッション悪用(session riding)は防げないため、出力エスケープ等の XSS 対策と常に併用する

### 暗号と資格情報

- **自前実装の禁止**: 暗号・認証・セッション管理を自作しない。フレームワーク・実績あるライブラリの実装を使う
- 転送は TLS を必須とし、保存する機微データは暗号化する
- パスワードは専用のパスワードハッシュ(計算コストを調整できる適応型ハッシュ)で保存する。可逆な形式・汎用ハッシュでの保存を禁止する

### セキュリティイベントの記録

- 認証失敗・認可拒否・検証失敗などのセキュリティイベントは記録する(logging-observability の様式に従う。秘匿情報は出さない)

### 悪用対策の責務分担

- アプリ層が担うのは L7 の悪用対策まで: レート制限・リクエストサイズ上限・タイムアウト・ページネーション上限
- ボリューム型 DDoS はアプリでは防げないため、エッジ(CDN・WAF 等)の責務として切り分ける

### secrets(成果物側)

- コードに secrets を書かない。ログに秘匿情報を出さない(logging-observability と接続)

### エージェントの裁量制限

- **セキュリティ敏感領域 — 認証・認可・secrets・CORS・脆弱性のある依存 — の変更は自律判断せず、必ずエスカレーションする**
- 敏感領域に触れる PR は、通常レビューに加えてセキュリティ観点のレビューを併用する

## トレードオフ

- **得るもの**: 既定が安全側に倒れ、エージェントが個別判断でセキュリティを弱めることを構造で防ぐ。検証・認可・ログの方針が他プリセットと接続して一貫した防御層になる
- **諦めるもの**: 敏感領域のエスカレーション必須により該当変更の自律性は下がる(意図した制約)
- **緩和策**: エスカレーション時は選択肢と pros/cons を添え、人間の判断コストを最小にする
