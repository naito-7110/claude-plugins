# feature-flags: フィーチャーフラグ運用

## 適用除外

main 常時リリース可能の運用をしない使い捨てリポジトリ・単発スクリプトには該当なし。継続開発するプロダクトにはすべて適用する。

## コンテキスト

main を常にマージ可能・リリース可能に保つため、未完成の機能はフィーチャーフラグの背後に置く(pr-granularity の積み上げ開発と merge-policy の前提)。一方で、所有者不明のまま放置されたフラグは負債となり、コードパスの組み合わせ爆発を招く。「フラグには必ず期限と所有者がある」状態を構造的に強制する。

## 決定

### フラグの形(4 属性必須)

すべてのフラグは一覧できる 1 箇所(レジストリ)で管理し、次の 4 属性を必須とする:

- `name`: 識別子(フラグを参照する全層で共有)
- `owner`: 所有者(削除に責任を持つ人)
- `expiresOn`: 期限日
- `description`: 目的

### 期限切れの強制

- **`expiresOn` を過ぎたフラグがレジストリに残っている場合、CI を失敗させる**。放置を構造的に不可能にする

### 値の所有

- フラグの値の定義は 1 箇所(配信側)が所有する。複数の層・コンポーネントに値の定義を重複させない(二重管理の禁止)

### リリース手順

- **フラグを削除する PR をもって機能の正式リリースとする**。削除 PR ではフラグ分岐の両側のうち新コードパスだけを残す
- リリースが通常のレビュー・マージフローに乗ることで、特別なリリース作業を作らない

### 粒度

- フラグによる出し分けは機能・画面単位を基本とする。細粒度の分岐を乱立させない

## トレードオフ

- **得るもの**: main が常にリリース可能・フラグの放置が CI red で構造的に不可能・リリースがレビューフローに乗る
- **諦めるもの**: フラグ登録・削除の手数が増える。実験的な A/B テストや段階的ロールアウトのような高度な配信は対象外(必要になった時点で別途検討)
