# authorization: 認可モデルの原則

## 適用除外

認証・認可を持たないリポジトリ(完全公開の読み取り専用ツール等)には該当なし。ユーザーごとに許可が異なる機能を 1 つでも持つなら適用する。

## コンテキスト

認可の要件(RBAC / ABAC / リソース単位)はプロジェクトごとに異なるが、責務分担を誤ると「見た目では隠れているが実行はできる」型の事故になる。モデルの選定はプロジェクトに委ね、責務分担の原則だけを固定する。

## 決定

### 認可の真実はサーバー(実行側)

- 認可の判定と強制は常にサーバー(実行側)で行う。**クライアント・UI 側の制御(ボタン非表示・ページガード等)はユーザビリティのためであり、防御ではない**
- UI の制御漏れがセキュリティ事故にならない構造を保つ(product-security と接続)

### 二層防御

サーバー側は 2 段階で弾き、境界の基準は「**対象データを読む必要があるか**」とする:

1. **境界層**: 対象リソースを読まずに判定できるもの(エンドポイント × ロール・システム権限)は、リクエスト処理の入口で弾く
2. **ユースケース層**: 対象リソースを読んで初めて分かるもの(所有関係・テナント越境・状態依存の拒否)は、ユースケース内で判定し「権限なし」カテゴリの失敗値で返す(error-handling と接続)

### 401 と 403 の使い分け

- **401**: 「誰か」を確立できない(未認証・セッション無効)。再認証で解決しうる
- **403**: 認証済みだが操作が許されない。再認証では解決しない

### モデルの拡張性

- 認可モデルの中核は要求(policy)ベースの構造に置き、ロールベース(RBAC)はその一実装として扱う。将来の属性ベース・権限ベースへの拡張を骨格の変更なしで受けられるようにする

### 判定のカプセル化

- **アクター × 権限 × 操作の対応は設計成果物(対応表)として明示**し、実装はそれを写像する**1 箇所にカプセル化された認可ポリシー**とする。ユースケースの中に判定式を直書きして散在させない
- リソースの状態・所有関係に依存する判定(ドメインロジック)もカプセル化の対象とする。ドメインの語彙で書かれた判定として切り出し、ユースケース層はそれを呼んで強制する
- **プレゼンテーション層とユースケース層は同じ判定を参照する**(重複実装しない)。層ごとに異なるのは関心だけ:
  - プレゼンテーション層: 「何ができるか」の照会(表示・導線の制御 = ユーザビリティ)
  - ユースケース層: 強制(拒否を失敗値として返す = 防御)
- 判定の真実が 1 箇所にあることで、UI の表示と実行時の強制がズレない(二層防御の 2 つの層が同じ対応表から導出される)

## トレードオフ

- **得るもの**: UI の制御漏れが事故にならない。二層の境界基準が機械的で、エージェントがどちらの層に書くか迷わない。判定の真実が 1 箇所(対応表 + カプセル)にあるため、表示と強制がズレず、対応表がそのまま仕様・テストの対象になる
- **諦めるもの**: 対応表とカプセル化された判定の維持コストがかかる(判定をその場に書く方が一時的には速い)
- **緩和策**: 認可に触れる変更はエスカレーション + セキュリティ観点レビュー必須(product-security の敏感領域)。散在した判定の回収より対応表の維持の方が安い、という前提を崩さない
