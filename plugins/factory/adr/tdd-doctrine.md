# tdd-doctrine: TDD ドクトリン(状態検証主義)

## 適用除外

自動テストを持たないリポジトリ(純ドキュメント・設定のみ等)には適用しない。プロダクトコードを新規に書き始めた時点で適用が始まる。

## コンテキスト

テスト戦略は古典派(デトロイト)とモック主義(ロンドン)の対立軸で語られがちだが、本ドクトリンはどちらの純粋形でもない。採用するのは「**検証は常に入力と最終的な出力・状態で行い、実装の内部動作(どのメソッドがどう呼ばれたか)に意味を見出さない**」という検証様式である。テストダブルの使用自体は禁止しない — 責務が適切に分割されていれば、依存の置き換えには意味がある。エージェントが大量のテストを書く前提では、この使い分けを機械的に判定できる形で固定する必要がある。

## 決定

### 第一原則: interaction 検証の禁止

- 検証は**最終的な Input → Output と状態変化**のみで行う。メソッドの呼び出し回数・順序・引数をアサートしない
- テストダブルを使う場合、その役割は入力の供給(stub)か状態の記録(fake)であり、「呼ばれたこと」の検証(mock)には使わない
- fake の最終状態(保存された内容)をアサートするのは状態検証であり、許容される

### 失敗するテストを先に書く

- red → green → refactor のサイクルで実装する。受け入れ条件を固定するテストから始める

### 層別の既定(具体はローカル ADR で定める)

- **ドメイン層**: 依存ゼロの純粋テスト。実物のみを使う(ここにダブルが必要になる時点で設計を疑う)
- **ユースケース層**: 対象以外の依存(リポジトリ・外部サービス等のポート)は stub / fake で置き換えてよい。ただし**ドメインオブジェクトは常に実物**を使う
- **UI 層**: 表示を担う部品はデータ供給側(HTTP 境界・facade 等)を stub し、「入力(プロパティ・状態)→ 描画結果・発火イベント」の状態検証で書く
- 層構成はプロジェクトのアーキテクチャに依存するため、**層別戦略の具体化はローカル ADR(アーキテクチャの ADR)の責務**とする。本ドクトリンが固定するのは検証様式(状態ベース)とダブルの使い分け基準のみ

### DB を伴うテスト

- 既定は**検証用の実 DB**を使う(インメモリ実装等での代替を禁止)。SQL の方言・制約・トランザクション挙動は実物でしか検証が意味を保たないため。テスト間はトランザクションのロールバック等で独立性を保つ
- 検証用の実 DB を用意できない場合(ライセンス制約等)は、**DB エンジンを偽装しない**。代わりに:
  1. ポート(リポジトリ境界)を fake にして、ユースケース層以上をテストする
  2. SQL・マイグレーション・クエリの検証は「実 DB でのみ実行するテスト」として分離し、実行環境が用意できるまで**検証できていない範囲として明示**する
- 判断軸は「**その代替で検証が意味を保つか**」。意味を保てない代替で緑を作るより、未検証範囲を可視化する方を選ぶ

### 禁止事項

- interaction 検証(呼び出し回数・順序・引数のアサート)
- 意味を保てない代替で緑を作ること(インメモリ DB での SQL 検証など)
- 既存テストのアサーションを弱める変更
- 「通すためだけのテスト」: アサーションが自明・実装の現状を写しただけ・受け入れ条件と対応しない

### 担保

セルフレビューとレビュアの検出観点に上記の違反(特に interaction 検証と「通すためだけのテスト」)を含める。

## トレードオフ

- **得るもの**: テストが公開された振る舞いだけに結合し、内部リファクタリングで壊れない。ダブル使用可否の判断が「何を検証しているか(状態か interaction か)」の 1 点に還元され、機械的に下せる。層別の具体はアーキテクチャに追従できる
- **諦めるもの**: mock による「協調の設計を駆動する」ロンドン派の利点は採らない。実 DB 前提の環境整備コストがかかる
- **緩和策**: 各実装の責務を最小に保って失敗時の特定精度を確保する。共有フィクスチャと最小のテストデータ構築ヘルパーでセットアップコストを抑える
