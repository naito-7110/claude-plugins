# tdd-doctrine: TDD デトロイト派ドクトリン

## 適用除外

自動テストを持たないリポジトリ(純ドキュメント・設定のみ等)には適用しない。プロダクトコードを新規に書き始めた時点で適用が始まる。

## コンテキスト

モックを多用するテストは実装の内部構造と結合し、リファクタリングのたびに壊れる負債になりやすい。エージェントが大量のテストを書く factory の前提では、壊れにくく意図の明確なテスト様式を規範として固定する必要がある。そこでデトロイト派(古典派)の TDD を採用し、モックの許可範囲を機械的に判定できる形で明文化する。

## 決定

### 基本原則

- **失敗するテストを先に書く**。受け入れ条件を固定するテストから始め、red → green → refactor のサイクルで実装する
- **状態ベースで検証する**。観測可能な出力と状態の変化をアサートする
- **自前のクラス・モジュールは実物を使う**(実コラボレータ)

### モック・スタブの許可範囲(プロセス境界のみ)

- 外部ネットワーク境界(HTTP など): 偽装してよい
- 時刻・乱数・ID 生成: 注入で差し替えてよい
- **DB はプロセス境界だが「実物を使う」側**。インメモリ実装・リポジトリ層のスタブで代替しない。検証用の実 DB を使い、テスト間はトランザクションのロールバック等で独立性を保つ

### 判断軸(モックか実物か)

1. **プロセス境界か** — 境界でなければ実物一択
2. **その代替で検証が意味を保つか** — 境界であっても、代替すると検証対象が消えるなら実物を使う。インメモリ DB で SQL を流しても、方言・制約・トランザクション挙動は検証できない

### 禁止事項

- 自前実装のモック化
- interaction 検証(メソッドの呼び出し回数・順序のアサート)
- 既存テストのアサーションを弱める変更
- 「通すためだけのテスト」: アサーションが自明・実装の現状を写しただけ・受け入れ条件と対応しない

### 担保

セルフレビューとレビュアの検出観点に上記の違反(特に自前クラスのモック化と「通すためだけのテスト」)を含める。

## トレードオフ

- **得るもの**: テストが公開された振る舞いだけに結合し、内部リファクタリングで壊れない。実 DB を通すことで、クエリ・スキーマの不整合を単体テスト段階で検出できる。モック可否の判断が 2 つの軸で機械的に下せる
- **諦めるもの**: 実 DB 前提のためテスト実行が重く、テスト用 DB の環境整備が必要になる(整備の具体はスタック事実・レシピ側の関心事)
- **緩和策**: 各実装の責務を最小に保って失敗時の特定精度を確保する。共有フィクスチャと最小のテストデータ構築ヘルパーでセットアップコストを抑える
