# pr-granularity: PR 粒度の原則

## 適用除外

なし。PR を作るすべての作業に適用する。

## コンテキスト

factory では人間のレビューがマージゲートであり、「人間がレビューできる PR」であることがスループットの生命線になる。並行するエージェント開発ではコンフリクトの最小化も効く。粒度の基準をサイズではなく関心事に置き、判断を機械的にできるようにする。

## 決定

### 分割基準は「関心事が単一で最小か」

- コミット・PR の分割基準は**関心事の単一性**であり、サイズ(行数・ファイル数)ではない
- **機械的な一括変更**(単一の変数名の一括置換・一括フォーマットなど、読んでいて苦にならないもの)は何ファイルあっても 1 PR でよい
- ただし**複数種の一括変更の混在**(複数の変数名の置換を 1 PR に混ぜる等)はレビュー負荷が跳ね上がるため、種類ごとに分割する
- **独立してレビュー・マージできる成果物が複数**含まれるなら、小さくても分割する(例: 独立した部品 3 つ → 3 PR)。分割案は issue コメントで提案・合意する

### サイズはシグナルとして使う

- サイズ超過(目安: 20 ファイル / 800 行)は「関心事が混ざっていないか」を見直すシグナル。単一関心だと説明できるなら超えてよい(PR の概要にその旨を明記する)

### 積み上げの作法

- 未完成の機能は feature-flags の背後に置き、main を常にリリース可能に保ったまま、成果物・レイヤーごとに PR を積む
- 分割した PR には依存順を明示する(先行 PR の番号を本文に書く)

## トレードオフ

- **得るもの**: レビュー時間の短縮と精度の向上・並行開発でのコンフリクト最小化・revert の容易さ
- **諦めるもの**: PR 数が増え、依存順の管理と積み上げの手数がかかる
- **緩和策**: 分割案の提案を仕様すり合わせ(spec-alignment)の必須アジェンダに含め、着手前に切り方を確定する
