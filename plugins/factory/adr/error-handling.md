# error-handling: エラーハンドリング原則

## 適用除外

対象は factory が生成する成果物のプロダクトコード。使い捨てスクリプトには縮小適用でよい(その場合も「予期される失敗と欠陥の区別」だけは保つ)。

## コンテキスト

エラーの表現方法を実装ごとの判断に任せると、失敗の分類・トランスポートへの変換・文言・ログが散在して揺れる。また「予期される失敗」と「実装の欠陥」の区別が曖昧だと、欠陥が握りつぶされて見えなくなる。エージェントが大量に書くコードでは、この区別を構造で強制する。

## 決定

### 予期される失敗は値で表現する

- 業務上起こりうる失敗(対象が存在しない・入力が不正・競合した等)は**戻り値(Result 型など)で表現し、例外で表現しない**
- 失敗値は次を持つ:
  - **安定コード**: `{domain}.{operation}.{code}` 形式の識別子
  - **意味カテゴリ**: 少数固定の分類(NotFound / ValidationFailed / Conflict / Forbidden / Unauthenticated / AlreadyModified / Unexpected 相当)。カテゴリの追加は本プリセットまたはローカル ADR の改訂を伴う
  - **構造化パラメータ**: 文言テンプレートに差し込む事実(名前・件数など)
- **人間向け文言を失敗値に持たせない**。文言の合成は表示層の 1 箇所で行う(i18n-copy と接続)

### ドメインの例外 = 実装漏れのシグナル

- ドメイン層の不変条件ガードは容赦なく例外を投げてよい。**その例外は「上位層の検証漏れ」のシグナル**である
- 上位層はガードに到達しないよう事前に検証する。**ドメインの例外を catch して失敗値に変換することを禁止する**(実装漏れが見えなくなるため)
- 到達してしまった例外は、境界の一元ハンドラが Unexpected(予期しない失敗)として処理・記録する(logging-observability と接続)

### トランスポートへの変換は 1 箇所

- 意味カテゴリ → トランスポート表現(HTTP ステータス等)の対応は**カタログ 1 箇所**に集約する。個別のハンドラ・コントローラでステータスを直書きしない
- 判定に迷うケースは「**エラーの主語**」で切り分ける: 主語がリクエスト内容(入力とサーバー状態の不整合)なら ValidationFailed、主語がリソースの現在状態(内容は妥当だが状態が許さない)なら Conflict。決まらないケースは仕様すり合わせで確定し、対応表に追記する

## トレードオフ

- **得るもの**: 失敗の分類と応答が全体で一貫する。予期される失敗と欠陥が構造的に区別され、欠陥の握りつぶしが起きない。呼び出し側が失敗値の列挙で応答の全体像を読める
- **諦めるもの**: カテゴリを少数固定にするため、収まらないエラーが現れたときは改訂手続きが要る。失敗値の定義・列挙の手数が増える
- **緩和策**: 迷いやすい判定は「主語」基準と対応表への追記で吸収し、カテゴリ自体の増殖を防ぐ
