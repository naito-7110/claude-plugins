# documentation: ドキュメント管理の原則

## 適用除外

なし。factory を設置したすべてのリポジトリに適用する(最小構成では地図とローカル ADR だけでもよい。ドメイン分割は漸進導入でよい)。

## コンテキスト

エージェント開発では文書がそのまま判断材料になるため、**古びた文書は無い文書より悪い**(エキスパートも人間も、古い記述を根拠に自信を持って誤る)。文書の置き場・所有・更新規律を憲法として固定し、鮮度を機械検証できる構造にする。

## 決定

### 文書の階層(どこに何が入るか)

| 層 | 置き場 | 中身 |
| --- | --- | --- |
| 憲法 | プリセット(プラグイン)+ `docs/adr/` | 原則と決定。改訂は改憲手続き |
| ドメイン知識 | `docs/domains/<domain>/` | 責務・ユビキタス言語・不変条件・公開契約・ドメイン決定 |
| 事実 | CLAUDE.md のマーカー節 | 検証コマンド等、リポジトリから導出された事実 |
| **地図** | `docs/factory/` | 「どこに何が入るか」のマニフェストと、**機械可読な所有マップ**(パス → ドメイン) |

作業状態(ジャーナル・台帳)は文書ではない(`.agents/`、gitignore)。この階層に無い場所へ規範・知識文書を增やさない(散在の禁止)。

### 同 PR 更新原則

- **コード変更によって古くなる文書は、同じ PR で更新する**。「あとで書く」を禁止する(文書だけの後追い PR は迷子になり、古い文書が判断材料に使われ続ける)
- 仕様すり合わせの段階で「この変更で古くなる文書」を特定し、更新を受け入れ条件に含める(spec-alignment のアジェンダ)

### 所有の明示

- 主要なコードパスは**いずれかのドメインに属する**ことを所有マップで宣言する。どのドメインにも属さないパスは検出・警告の対象とする(所有者のいないコードは文書もレビュー責任も宙に浮く)
- 所有マップは機械可読な形式で `docs/factory/` に置き、スキル(groom / work / huddle)が「この diff はどのドメインの管轄か」を機械的に引けるようにする

### 公開契約の明文化

- ドメイン間の境界(API・イベント・共有スキーマ)は該当ドメインの契約文書に明文化する
- 実装と契約文書の乖離を発見したら、どちらかを直すことを同 PR または即時の issue 化で行う(乖離の放置禁止)

### 検証の機械化

- 文書構造(必須ファイルの存在)・地図の形式・所有マップと実パスの整合は、bin による機械検証を CI に乗せる。人間の注意力に依存させない

### 書き方

- 1 文書 1 目的。結論(決定・責務)から書き、経緯は後ろに置く
- 切れやすい外部リンクより内容の蒸留を優先する(憲法は外部リンク全面禁止 — adr/README の形式規約)

## トレードオフ

- **得るもの**: 文書が常に判断材料として信頼でき、ドメインエキスパート(huddle)の品質基盤になる。文書更新がレビューの一部になり、暗黙知の属人化が減る
- **諦めるもの**: すべての変更 PR に文書更新の確認コストが乗る。所有マップの維持が必要になる
- **緩和策**: 検証を bin で機械化し、確認コストを「警告に答える」だけに圧縮する。ドメイン分割は 1 ドメインから漸進導入できる
