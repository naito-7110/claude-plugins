# merge-policy: マージポリシー

## 適用除外

なし。factory の運用に乗るすべてのリポジトリに適用する。前提: main へのマージは本番デプロイではなく、本番反映は別のゲート(タグ push 等)で人間が行う。

## コンテキスト

「マージは常に人間」だと夜間・並列のスループットが人間のレビュー待ちで頭打ちになる。main ≠ 本番であれば revert は安く、fail-closed の錨をデプロイ側に置ける。そこでマージゲートを issue ごとの可変ポリシーにし、既定は人間・例外的に agent マージを許す二レーン制にする。

## 決定

### 二レーン制(既定は人間マージ)

- **既定は人間マージ(fail-closed)**。ラベル `merge:agent` が付与された issue に紐づく PR のみ、agent がマージまで実行してよい
- 本番反映(タグ push・デプロイ)は常に人間。agent マージが増えてもこの最終ゲートは動かない

### merge:agent の付与プロセス

- 付与は**仕様すり合わせの場に限定**する: AI が本ポリシーを基に「このタスクは agent マージ可か」を提案し、**人間が承認して Ready 化と同時に付与**する
- トリアージ・起票時は提案コメントまで(ラベルは付けない)。**無人セッションは付与・変更ともに禁止**
- これにより「人間がレビューしていない merge:agent」は構造的に存在しない

### agent マージの実行条件(すべて AND)

1. **CI が green** — 通常のプロダクト CI(ビルド・テスト)と、**PR↔issue 整合チェック**(関連 issue のラベル・鮮度・受け入れ条件の存在を検査する check)の**両方**。後者は「ラベリングされていない issue の実装がコードに入ることを防ぐ最終ライン」の責務を持つ
2. **実装コンテキストを共有しない別コンテキストのレビュアが green**(実装セッション自身のセルフレビューでは代替できない)
3. **鮮度**: `merge:agent` 付与後に issue 本文・受け入れ条件が変更されていない(変更されていたら付与時の承認は無効とみなし、人間レーンへ降格)
4. レビュアから要対応の指摘が出た場合は、**再ラベリングして人間レーンへフォールバックする**: `merge:agent` を外し、指摘の要約と判断材料を issue コメントに記録する(議論を issue に積み重ね、人間が解消後に再付与できる状態を保つ)

### 失格条件(常に人間マージ)

- セキュリティ敏感領域: 認証・認可・secrets・CORS・脆弱性のある依存(product-security の敏感領域と同一集合)
- 破壊的な外部 API 変更・DB スキーマ変更
- **関連 issue に明記されていない依存(ライブラリ・ツール)の追加**。依存の追加が見込まれる場合は仕様すり合わせで揉んで issue に明記する(spec-alignment のアジェンダ)。やむを得ず実装中に増えた場合は、選定理由(pros/cons)と安全性の確認(supply-chain-security の観点: 既知脆弱性・クールダウン)を PR に明記した上で人間マージとする
- 上記に実装中に触れることが判明した場合も、その時点で再ラベリングして人間レーンへ降格する

### 強制手段は書かない

- 本ポリシーの機械的な強制方法(hook・required check・approve 等)は実装詳細であり、本 ADR には書かない。ポリシーは強制手段の選択に依存せず成立する

## トレードオフ

- **得るもの**: 夜間・並列のスループット(人間の朝の仕事が「事前レビュー待ち行列の消化」から「マージ済み一覧の事後レビュー + 必要なら revert」に変わる)。ラベル 1 枚で issue 単位に信頼度を調整でき、運用しながら agent レーンを段階的に広げられる
- **諦めるもの**: 別コンテキストレビュアの体制整備が前提になる。事後レビューと revert を許容する運用規律が要る
- **緩和策**: 失格条件と付与プロセスの人間ゲートにより、リスクの高い変更は最初から agent レーンに乗らない
