# performance: パフォーマンス原則

## 適用除外

なし。すべてのプロダクトコードに適用する。数値目標(レイテンシ・スループット)はここでは定めず、プロジェクト側(ローカル ADR・仕様)で定める。

## コンテキスト

性能問題は後から直すほど高くつく一方、根拠のない先回り最適化はコードを歪める。エージェントには「常に守る事故防止の下限」と「どこまで最適化するかの判断基準」の両方が要る。前者を構造で強制し、後者を計測に紐づける。

## 決定

### 常に守る下限(事故防止)

- **上限なき読み取りの禁止**: 一覧・検索・集計は必ずページネーションまたは件数上限を持つ。無制限の全件取得をコードパスに作らない
- **外部 I/O の時間と失敗の規律(タイムアウト必須・リトライの作法)は resilience プリセットを正準とする**。無限待ち・無秩序なリトライのコードパスを作らない
- **N+1(ループ内の逐次 I/O)をレビューの固定観点にする**。まとめ取り・バッチ化を既定とし、N+1 を許容する場合は理由を PR に明記する

### 最適化は計測駆動

- **計測なき最適化の禁止**。可読性を犠牲にする最適化 PR は、before/after の計測結果を添える
- 逆も同じ: 「遅い気がする」だけで実装を複雑にしない。まず計測する(logging-observability の計測基盤と接続)

### SLO(性能要件は仕様の一部)

- プロジェクトは主要なユーザー体験・処理について **SLO(レイテンシ・エラー率・可用性などの定量目標)をローカル ADR で定める**。仕様すり合わせでの性能目標値は SLO から導出し、その場の感覚で決めない
- 大量データ・高頻度アクセスが想定されるユースケースは、**仕様すり合わせで目標値(データ量・応答時間の目安)を確定してから**実装する(spec-alignment と接続)
- SLO 未定義の領域・目標値のないユースケースについて「十分速いか」の議論をしない(下限さえ守れば要件駆動で足りる)
- 定量指標は SLO に限らず有用だが、**意思決定に使う指標だけを構築する**(測るために測らない)。指標の追加は理由とセットでローカル ADR に記録する

## トレードオフ

- **得るもの**: 無限読み・無限待ち・N+1 という「後から直すのが最も高くつく事故」を構造的に防ぐ。最適化の判断が計測に紐づき、憶測での複雑化が起きない
- **諦めるもの**: ページネーション・タイムアウト・計測の一手間が常にかかる。数値目標を持たないユースケースでは「最速」は目指さない(意図した割り切り)
- **緩和策**: 上限・タイムアウトの既定値はスタック事実・レシピ側で用意し、実装ごとの判断を減らす
