# git-workflow: git ワークフローの原則

## 適用除外

なし。factory の運用に乗るすべてのリポジトリに適用する。

## コンテキスト

ブランチモデル・コミット規約・履歴の扱いが揺れると、並行するエージェント開発では衝突と追跡不能な履歴を生む。人間ゲート(レビュー・マージ・タグ)が機能するためには、変更の流れが 1 つの型に固定されている必要がある。

## 決定

### トランクベース

- **main が唯一の長命ブランチ**。develop・release などの長命ブランチを作らない
- main は常にリリース可能に保つ(未完成は feature-flags の背後へ)
- 本番反映は **tag push**(merge-policy のデプロイゲート)。バージョンはセマンティックバージョニングとする

### 作業ブランチ

- 作業ブランチは issue に紐づく短命ブランチとし、命名は `agent/issue-<n>-<slug>`(人間の作業も同型の `<type>/...` 短命ブランチ)
- base は常に main。PR を積む場合は依存順を PR 本文に明示する(pr-granularity)
- 並行作業は git worktree で分離する(`.worktrees/`、gitignore 済み)

### コミット

- conventional commits(type + scope 必須)、メッセージは英語
- type は次に固定する: `feat` / `fix` / `docs` / `style` / `refactor` / `perf` / `test` / `chore` / `hotfix`。追加・変更は本プリセットの改訂で行い、全プロジェクトで揃える
- 意味のある最小単位でコミットする。「作業一時保存」的なコミットを PR に残さない

### マージと履歴

- **main への直 push・force push を禁止**する(hook・ブランチ保護で機械的に強制)
- PR のマージ方式は **squash merge に固定**する: PR = 関心事が単一(pr-granularity)なので 1 コミット化が自然で、conventional な PR タイトルがそのまま履歴になる。**マージコミット・rebase マージは使わない** — 細かいコミット履歴に意味があると感じたら、それは PR を分割するシグナル(pr-granularity)
- 作業ブランチの main への追従は **rebase** で行い、取り込みのマージコミットを作らない(ローカル・リモートを問わず、マージコミットを履歴に残さない)
- PR ブランチは作者のみが書く前提とし、main 追従のための rebase + force push を許容する。複数人が共有するブランチでは履歴改変をしない

### 追跡可能性

- すべての PR は issue に紐づく(`Closes #N` / `Refs #N`)。issue に紐づかない変更を main に入れない(#17 の整合チェックが強制)
- revert は通常の PR フローで行う(main ≠ 本番なので revert は安い、が merge-policy の前提)

## トレードオフ

- **得るもの**: 変更の流れが 1 本化され、履歴が PR 単位で読める。並行エージェントの衝突が短命ブランチ + worktree で最小化される。ブランチ運用の判断が不要になる
- **諦めるもの**: 長期の大規模変更を 1 ブランチで抱える運用はできない(flags + 積み上げ PR で刻む前提)。squash 既定のため PR 内の細かいコミット履歴は main に残らない
- **緩和策**: 細かい履歴が要る場合だけ merge commit を選択できる余地を残す
