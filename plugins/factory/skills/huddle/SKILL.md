**対話専用**(huddle は仕様確定の場の一部 = 人間ゲート)。**発動条件**: 変更の影響が複数ドメインに跨る見込みのときだけ(単一ドメインなら通常の groom で足りる — 過剰協議はコストの無駄)。

## 1. 発動判定

- 対象 issue から変更が触れそうなパスを推定し、`.factory/ownership.yml` で管轄ドメインを引く
- **管轄が 2 ドメイン以上** → huddle 発動。1 以下 → 通常の /factory:groom へ(本スキルは終了)

## 2. エキスパート定義の確保

参加ドメインごとに、対象リポジトリの `.claude/agents/<domain>-expert.md` を確認し、**無ければ `docs/domains/<domain>/` から生成する**(プラグインは生成器、人格はリポジトリ側)。定義に必ず含めるもの:

- 責務・ユビキタス言語・不変条件(README から)と公開契約(contracts から)
- **材料の制約**: 自ドメインの文書(`docs/domains/<domain>/`)と所有パス(ownership.yml の自分の paths)のコードだけを読む。**他ドメインの内部・全体コンテキストを読まない**

あわせて**モジュール全把握エージェント**(横断視点)を用意する: 所有マップと依存の俯瞰だけを材料にし、どのドメインの内部にも入らない。

## 3. 第 1 相: 独立評価(コンテキスト非共有)

- **同一の設問**(変更内容 + 回答フォーマット)を各エキスパート + 全把握エージェントに **Task で並列に**投げる。互いの存在も評価も知らせない(アンカリング防止)
- 回答は構造化フォーマット固定:

```markdown
## <domain> の影響評価
- 影響: あり / なし(なしの場合も根拠を書く)
- 依存する契約: <自ドメインの contracts のうち、この変更に関係する項目>
- 必要な変更: <自ドメイン内で必要になる変更の列挙>
- 懸念: <リスク・不明点>
- 規模感: S / M / L(根拠 1 行)
```

## 4. 突き合わせ(矛盾の機械的検出)

全評価を突き合わせ、次を列挙する:

- **二重申告**: 同じ変更・同じ責務を複数ドメインが「自分の管轄」と主張している
- **押し付け合い**: 変更が必要なはずの箇所(全把握エージェントの俯瞰・設問の要求)を、どのドメインも管轄と言っていない
- **契約の不一致**: あるドメインが依存すると言った契約を、提供側ドメインが変更対象と認識していない

## 5. 第 2 相: 議論(結論サマリのみ共有)

- 矛盾点ごとに、当事者のエキスパートへ**相手の結論サマリだけ**を渡して反論・調整を求める(生の評価全文・全体コンテキストは最後まで渡さない)
- 本スキル(PM)が収束を判定する。**収束しない矛盾は捏造せず人間へ**(`AskUserQuestion` で裁定を仰ぐ — 境界の裁定は domain-partitioning の再編判断につながることが多い)

## 6. 記録と接続

- **協議の全過程**(各独立評価・検出した矛盾・調整の経緯)を issue コメントに残す(耐久状態)
- 収束結果を **「ドメイン別影響分析(YYYY-MM-DD huddle で確定)」節として issue 本文へ書き戻す**(huddle は仕様確定の場の一部であり、本文編集権は groom と同じ扱い)
- そのまま /factory:groom の必須アジェンダへ接続する(影響分析を前提に受け入れ条件・分割案を確定)。境界の裁定で所有マップの変更が要る場合は /factory:domains へ

## 禁止事項

- 第 1 相での評価の相互共有・エキスパートへの他ドメイン文書/全体コンテキストの供給
- 収束の捏造(不一致を残したまま「合意」としない — fail-closed で人間へ)
- コードの変更・merge:agent の付与(仕様の場であり実装の場ではない)
