**対話専用。無人セッションでは絶対に実行しない。** 改憲は人間ゲートであり、**`AskUserQuestion` での明示的な採択なしに ADR ファイルを書かない**。

## 手順

### 1. 入力の特定

- エスカレーション issue(「ADR 候補の発見」)からの誘導なら `gh issue view <n> --comments` で経緯・選択肢・pros/cons を読む
- 人間の直接提案なら、論点を聞き取って同じ形(選択肢 + trade-off)に整理する

### 2. 現行憲法の確認

- プリセット(`${CLAUDE_PLUGIN_ROOT}/adr/` — README の索引から関連分)とローカル `docs/adr/` を読み、**既に答えがないか・矛盾しないか**を確認する
- 既に答えがある場合は該当 ADR を示して終了する(改憲不要)。矛盾する場合は「上書き(Overrides)」として扱う

### 3. 三層の振り分け

持ち込まれた内容を三層に振り分け、行き先を確定する:

- **事実**(検証コマンド・ビルド手順・環境の実態)→ ADR ではない。CLAUDE.md のスタック事実へ案内する(/factory:init の再実行)
- **スタック非依存で他プロジェクトにも効く原則** → ローカル ADR として採択しつつ、**プリセット候補**としてプラグインリポジトリへの追記 PR を提案する(中央蓄積のフライホイール)
- **プロジェクト固有の決定(技術選定を含む)** → ローカル ADR。技術選定は理由・代替案つきで正当な ADR である(三層構造 — 拒否しない)。プリセットの規定を覆す場合は frontmatter に `Overrides: <slug>` を宣言する

### 4. 論点整理と対話

決定内容・検討した代替案・トレードオフを `AskUserQuestion` で 1 論点ずつ確定する。選択肢には必ず trade-off を添える。

### 5. 人間の裁可

採択 / 棄却 / 見送り を人間が明示的に決める:

- **採択** → 手順 6 へ
- **棄却** → 理由を記録する価値があれば status「棄却」の ADR として残す(「なぜやらないか」も憲法の一部)。残さない場合はエスカレーション元 issue にコメントで理由を記録する
- **見送り** → エスカレーション元 issue に判断材料を記録して閉じる(将来の再提案に備える)

### 6. ADR ファイルの生成・改訂

`docs/adr/NNNN-<slug>.md`。採番は既存ファイルのスキャンで最大番号 + 1。形式:

```markdown
---
Overrides: <slug>   # プリセットを上書きする場合のみ
---

# NNNN: <決定を一文で表すタイトル>

- ステータス: 提案 | 承認 | 廃止 | 棄却
- 日付: YYYY-MM-DD

## コンテキスト
<なぜこの決定が必要になったか。エスカレーション元 issue があれば #N>

## 決定
<採用する方針と判断基準。検討した代替案と退けた理由>

## トレードオフ
<得るもの・諦めるもの・緩和策>
```

- **外部リンクを本文に貼らない**(参照資料は内容を蒸留する)。スタック固有の内容はローカル ADR では許容される
- **改訂・廃止は歴史を消さない**: 旧 ADR は本文を書き換えず、ステータスを「廃止」にして置き換え先(新 ADR 番号)を記載する。新 ADR 側にも置き換え元を記載する(supersede の双方向記録)

### 7. PR 作成と承認

- 通常の PR フローで提出する(ブランチ → PR → **人間のマージ = 改憲の最終承認**)。エスカレーション元 issue があれば `Refs #N`
- プリセット候補と判定した場合は、プラグインリポジトリへの追記 PR(または issue)を別途起こすことを人間に提案する

## 禁止事項

- 無人実行
- `AskUserQuestion` での採択なしに ADR ファイルを書く・改訂する
- 旧 ADR の本文書き換えによる改訂(supersede を経ない歴史の消去)
- 事実(検証コマンド等)の ADR 化

## 備考

- 採番の並列衝突(複数の改憲が同時に走る場合)は Phase 1 では対話専用のため実害がない。PR 時点の再スキャンで解消する
