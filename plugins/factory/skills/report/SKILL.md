「不在の間になにが起き、いま人間がなにを判断すべきか」を 1 箇所で読めるようにする。**夜専用ではない**(レーン統一 — #4)。通知は issue コメントのみ(外部サービスを使わない)。

## 手順

### 1. 対象期間の特定

運用 issue(「factory operations」— 無ければ作成する)の最後の report コメントの時刻以降を対象とする。初回・見つからない場合は直近 24 時間。

### 2. 収集(すべて gh と factory バイナリから)

- **成果**:
  - 期間内に作成された agent PR(open)= レビュー待ち一覧
  - 期間内に **agent がマージした PR(merge:agent)= 事後レビュー一覧**。各行に diff へのリンクと revert 導線(squash コミットの `git revert <sha>` 手順)を付ける
- **要判断**:
  - 未解決の `needs-human` 一覧(各 issue の最新エスカレーションコメントの要旨つき — 人間がコメントを開かなくても判断の種類が分かる粒度)
  - 制動条件の発動有無(orchestrate のサイクル報告から: PR 滞留 / エスカレーション滞留)
- **滞留と健康**:
  - 人間レーンの In Review 滞留数と最古の滞留日数
  - `factory flags check --warn-days 14` の期限接近フラグ
  - `factory docs verify` の警告(未所有パス・重複所有)

### 3. 報告

運用 issue に **1 コメント**で報告する(1 実行 = 1 コメント)。構成:

```markdown
## factory report(YYYY-MM-DD HH:MM)

### 成果(期間: 前回報告以降)
- レビュー待ち: #N ... / agent マージ済み(事後レビューを): #N ...(revert: `git revert <sha>`)

### 要判断
- needs-human: #N — <要旨> ...
- 制動: <発動なし / PR 滞留で配車停止中 ...>

### 滞留と健康
- In Review: n 件(最古 n 日) / 期限接近フラグ: ... / docs 警告: ...
```

### 4. 沈黙しない

期間内に何も起きていなくても「実行なし・変化なし」とその理由(制動中・Ready 枯渇・前提チェック不成立など)を必ず報告する。**報告が無いことと問題が無いことを区別できるようにする**。

## 禁止事項

- issue 本文の編集・ラベル操作(報告に徹する。判断は人間)
- 外部サービスへの通知
